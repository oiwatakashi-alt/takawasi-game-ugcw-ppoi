# ゴールループ統制論 — ループの上の管理層(旧05_GOVERNANCE。v2.0で11へ移設)

00〜10はループの中の規律。この文書は**ループの外**に立つ人間(あなた)の管理論。
対象が変わる: コードでもターンでもなく、走行(run)とポートフォリオを管理する。

## 第一命題: 労働は自動化されたが、統制は自動化されていない

ゴールループが肩代わりするのは生産であって判断ではない。人間の仕事は4つに収縮する:
**ゴール設定 / 検収 / 資源配分 / 例外処理**。この4つ以外(進捗の監視、コードの逐次
レビュー、途中の指示出し)をやり始めたら、それはループの失敗かあなたの過干渉の
どちらかであり、どちらでも設計の修正対象。

## 第二命題: 報告は読むが信じない——統制は外形指標に載せる

エージェントの自己申告(完了報告・自己評価)は系統的に過信側へ歪む。統制の計器は
自己申告と独立した外形に置く:

| 計器 | 取り方 | 何の兆候か |
|---|---|---|
| 変更層の分布 | git logの触ったファイル種別 | 表示層偏重=深さの停滞 |
| テストの実質増分 | テスト数と対応機能の突合 | 非増加=検証の空洞化 |
| STATE鮮度 | STATE記載とgit HEADの乖離 | 乖離=台帳の腐敗 |
| 予算消化率 | フェーズ消費ターン/目安 | 超過=ゴールの切り方が悪い |
| 同一失敗回数 | STATEの失敗カウンタ | 3回=場当たりループ |
| 検収待ち行列 | 完了済み未検収フェーズ数 | 滞留=ボトルネックは人間側 |
| **機構の消費率** | 作った機構の出力がLOGの報告消費欄に現れる頻度 | ゼロ=誰も読まない機械の量産(儀式的に完璧なターンは進捗に見える——見た目に騙されない) |
| **自己参照率** | 直近ターンのうちワークフロー自身の整備が占める割合 | 過半=ループがループを磨いている(製品が進んでいない) |
| **総和密度トレンド** | 成果物全体の機械計測値(UI要素数・同時演出数・イベント頻度等)のバッチ間推移 | 上がり続ける=品位線の劣化。差分検証は総和の過剰を検出できない |

このうち機械算出できるものは `tools/check.sh` が出す。週次点検はスクリプト実行から
始め、手で読むのは黄・赤が出た走行だけにする。報告と計器が食い違ったら計器が正しい。

## 第三命題: 介入点は境界にのみ置く

ターンの途中に人間が入るならループを回す意味がない。介入は3種に限定する:

1. **検収**(フェーズ境界): 効果レベルの判定。人間にしかできない唯一の工程
2. **継続判断**(予算超過時): 続行/フェーズ分割/返上の三択
3. **例外**(エスカレーション時): 3回失敗・破壊的操作の要求・前提の崩壊

それ以外の中間介入は原則しない。したくなったら、その衝動自体を「ゴール文か
テンプレの欠陥の発見」として記録する——直すのは走行ではなく仕様の側。

## 第四命題: ボトルネックは常に人間の検収能力である

- **WIP制限**: 同時走行数の上限は「あなたが検収を回せる数」。検収待ちが2件溜まったら
  新規走行を止める
- **検収コストの圧縮は完了条件で強制済み**: 検収パック(01_PLAN)の提出をフェーズ完了
  条件にした。人間の検収は「証跡を見て効果を判定する」に収縮する
- **証跡は自己申告として扱う**: 月1回程度、証跡の再現手順を実際に叩く抜き取り検査を
  入れる。再現しない証跡が1件でも出たら、その走行の全証跡を疑いフェーズを差し戻す
- **best-of-N は検収の経済で考える**: 判定が安い領域でだけ使う

## 第五命題: 既定の倒し方は運転嗜好で選ぶ(v2.0改訂)

旧版はfail-stop一択だったが、実走行でオーナーの運転嗜好(候補がある限り継続)と
即座に衝突した(CHANGELOG v2.0)。既定は00_MISSIONの運転設定で選ぶ:

- **fail-stop**: 疑義・予算超過・人間不在——迷ったら止まる。走りすぎの損失
  (誤った土台への積み増し)は巻き戻しでも消えない、という側に賭ける
- **継続**: 候補がある限り[下書き]タグで進み、検収で遡及確定する。止まりすぎの損失
  (無人時間の空転)を嫌う側に賭ける。巻き戻しはcommit単位で可能なことが前提

どちらでも不変: ラチェット(緩和=人間のみ)・不可逆操作の停止・同一失敗3回の停止。

## 第六命題: 二階のループを回す——走行の失敗はテンプレの改版材料

- 走行が失敗・脱線したら「テンプレのどの規則が機能しなかったか」を特定し、
  テンプレ側を改版する(このフォルダをgit管理し、改版ログを残す)
- 改版は観測された失敗を根拠にのみ行う。予防的な規則追加はしない
- 同じ失敗が2つの異なるプロジェクトで出たら、テンプレの問題と推定する
- 逆も回す: ある走行でうまく機能した運用はテンプレに昇格させる
- **規則は書かれただけでは実行されない**: 散文の規則は勢いに負けて静かに脱落する。
  安定して機械的になった規則は計器(check.sh等)かブリーフ注入に降ろすのが
  改版の最優先経路(実行される規則は発火失敗を持たない)

## 第七命題: ポートフォリオで見る

|---|---|---|---|---|---|
| [ ] | | | | | |

運用: 緑は触らない。黄は計器を確認して原因をSTATEで特定。赤は停止して返上を受ける。

### 渋滞プロトコル(ループが多重に煮詰まったとき)

兆候: 検収待ち2件以上/複数走行で黄・赤が同時点灯/humanops openが週をまたいで減らない。

1. **止める**: 新規走行の投入を停止
2. **排水する**: 全走行のDASHBOARD.htmlを開き、人間操作を一括処理する。渋滞の原因は
   ほぼ常に人間側の待ち行列——人間の30分の排水が、どの走行の改善よりも利回りが高い
3. **検収は古い順に一括処理**(古いものほど文脈が腐り、巻き戻しコストが複利で増える)
4. **再開は価値順に1本ずつ**

## 既知の病理図鑑(監査番の診断補助。v2.2)

LLMの故障はモデル横断で同型に出る(Sonnet/Fable/Grokの3機で同一クラスを実測)。
文言の強さは対策にならない——強く書かれた散文契約は43ターン中23ターンで無言脱落し、
同じ契約の機械検査成分は34/34で守られた。対策の階層は
**文言(生まれたて)→注入(brief)→ゲート(機械検出)→構造除去(生息地をなくす)**で、
症状が機械検出可能だと分かった病気はゲートに昇格させる義務を持つ。

書式は診断形(この観測が出たらこの病気)。訓戒形(〜するな)では監査に使えない。

| 病名 | 症状(観測可能なsignature) | 検出器 | 階層 |
|---|---|---|---|
| 早停め病 | 停止宣言時に非blockedの残課題・候補・発見が残存 | tools/stop_check.sh(棚卸し強制) | **ゲート済み** |
| 証跡なし完了病 | 「直した/完了」に対応する新規証跡がない | check.sh証跡鮮度+監査番の抜き取り | 半ゲート |
| 台帳腐敗病 | STATEとgit/残課題台帳の矛盾 | check.sh STATE鮮度 | **ゲート済み** |
| 無言脱落病 | 規則が守られなくなるが宣言がない | 予算脱落の宣言義務(00_MISSION)+LOG検査 | 注入+文言 |
| 化石限界病 | 「できない」リストの項目が後の波で実装される | 能力台帳+プローブ要求(07_KNOWLEDGE) | 文言 |
| 到達可能性配分病 | blocked隣接の主軸が飢え、届く軸だけ消化が進む | 軸生成工程の飢え軸計数(消化実績の軸別内訳が入力) | 半ゲート |
| 加算単方向病 | 削除・退役の記録ゼロのまま候補だけ増える | 監査番の削除指名義務+消費率計器 | 半ゲート |
| 機構自己参照病 | ターンの過半がワークフロー自身の整備 | ドリフト指標の自己参照率 | 半ゲート |
| 許可待ち病(早停め亜種) | 非blocked燃料が残るのに境界で「続けてよいか」と聞く(答えが行動を変えない質問) | CLAUDE/AGENTS核の継続既定(規則5)+報告形式への置換 | 注入 |
| 収束探索病 | 探索が棚卸しと既存作比較に限定され、燃料枯渇=オーナー起点投入待ちになる | 軸生成工程+stop_checkの空振り証跡要求 | **ゲート済み** |
| 段階不一致検証病 | 検証の厳密さが段階・不可逆性と無関係に一律(MVP前に磨き段階の検証を要求等) | 監査番で検証コストと段階の突合 | 文言 |
| 下流貼り紙病 | 構造欠陥にインスタンス側のパッチを貼り、テンプレ原本を直さない | 同型修正2回目でテンプレ改版を強制(09_LEARNINGS経路) | 文言 |

運用規則:
- 行の追加は観測された失敗の証跡つきのみ(予防的追加はこの表でもしない)
- 構造除去で生息地が消えた病気は退役させ、改版ログに記録する
- この表の肥大はそれ自体が敗北条件(中央モード表の裏口回帰の兆候として扱う)

## 実装の最小形

週次15分で: (1)各走行の `git log --oneline` と `04_STATE.md` を斜め読みして計器の色を
付ける (2)検収待ちを処理する (3)テンプレ改版候補があればこのフォルダに反映する。
管理のための管理(ダッシュボード自動化等)は、走行3本超で手が回らなくなってから作る
(00_MISSIONの機構関門はここにも適用——消費者と退役条件を言えない管理機構は作らない)。

## この文書の敗北条件

- 介入3種で足りない事例が繰り返し出る → 第三命題の改版
- 計器が緑のまま失敗した走行が出る → 第二命題の計器追加(失敗の観測を根拠に)
- WIP制限が理論倒れで、実際は検収をためても回る → 第四命題の緩和
