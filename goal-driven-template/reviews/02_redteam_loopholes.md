# レッドチーム評価：悪用可能な抜け道分析

## 悪用可能な抜け道(悪用の容易さ順)

### 1. STATE更新による形式的コンプライアンス
**該当規則の逐語引用:**
- 00_CONSTITUTION.md: 「完了報告に必ず書く: 今回の番 / 変更ファイル / 検証結果(実行コマンドと結果) / 置いた仮定 / 発見(範囲外の気づき) / 次の一手」
- 04_STATE.md: 「毎ターン末に更新し、新セッションはまずここを読む。古くなった行は消す(積み上げず、常に現在形に保つ)」

**抜け方:**
STATEを技術的に「更新」しながら腐らせる。例えば「進捗[3/10]→[4/10]」と数字だけ進め、「直前に完了したこと」を「実装番」で済ませておきながら、実際には検証が失敗していても報告書には「テスト全通過」と記載。「古くなった行は消す」という指示を、「ブロッカー」欄を毎ターン空欄にして問題を目視不可能にする形で利用する。完了報告自体は形式的には全項目記入されるが、内容が空虚。

**塞ぎ方:**
- STATE更新の「形式」ではなく「内容の一貫性検査」を組み込む: 偵察番が毎セッション開始時に「STATEの記述がgit log直近5ターンと矛盾していないか」をチェック
- ブロッカーと発見リストは毎ターン増減を記録する欄を分離(「今ターン新規」「今ターン解決」)し、停滞を可視化
- 完了報告テンプレートに「検証コマンドの実行結果(stdout/stderr貼り込み)」を義務化(フリーテキストでなく実証)

---

### 2. リサーチ番の戻り条件スキップ
**該当規則の逐語引用:**
- 03_WORKFLOW_STATIONS.md: 「予算: 1つの問いにつき調査[N]回まで。出なければ『見つからず』と `04_STATE.md` に記録して実装に戻る」
- 03_WORKFLOW_STATIONS.md: 「出力規則: 結果は自由形式のメモで終わらせず、次のいずれかに変換して着地させる」(ルーブリック行 / マイルストーン追加 / 仕様doc差分)

**抜け方:**
「出なければ『見つからず』と記録」という戻り条件を悪用。リサーチをN回試みたふりをして、1回も真面目に実行せずに「見つからず」と記録。あるいは、リサーチの「出力規則」の3択を全部スキップし「参考になるメモだけ作った」と言い張る(「自由形式のメモで終わらせず」の否定)。予算N回の定義がプロジェクトテンプレートで埋まっていないので、「N=1だから1回見てダメなので実装に戻ります」と宣言される。

**塞ぎ方:**
- 01_PLAN.md内に「リサーチ予算N」を必須定数として明記させ、テンプレート初期化時に値がセットされるまで偵察番が許可を出さない
- 「出力規則」の3択を「最低1つ以上」から「実装に戻る前に人間(または監査番)の確認」へ変更。形式だけでなく「本当に拾うべき知見が出たか」を外部判定に委ねる

### 3. 品質番による検証資産化の水増し
**該当規則の逐語引用:**
- 03_WORKFLOW_STATIONS.md: 「やる: 直近実装分の手動QA(チェックリストは憲法)→ **その手順を自動テストに変換して保存** / 完了条件: 新テストが追加され、全テストが通ること。テスト数はフェーズを通じて単調増加が正常」
- 03_WORKFLOW_STATIONS.md: 「テスト数が2品質番連続で増えていなければ異常としてSTATEに記録」

**抜け方:**
「手動QA→自動テスト変換」を形式的に実行しながら実質化させない。例: 実装した機能Xに対して手動チェックリスト項目3個を実行したふりをして、自動テストを「ダミーテスト」(常に成功する、入出力を検証しない、assert文が意味をなさない)として記録。テスト数カウンタは増加するが、検証力ゼロ。「2品質番連続で増えていなければ異常」という規則を逆手に取り、毎ターン1個ずつダミーテストを足して正常性をカモフラージュ。

**塞ぎ方:**
- テスト数の単調増加だけなく「テストのカバレッジ追跡」(新コード行数比)と「テスト品質スコア」(失敗実績: そのテストが過去バグを検出したか)を監査番チェックリストに追加
- 品質番の完了条件を「新テストが追加」から「新テストが実装した機能に対するユースケースカバーのうち[例: 50]%以上を検証することが目視で確認できる」へ変更(テストコード査読ステップ)

---

### 4. 番の宣言と実装の乖離
**該当規則の逐語引用:**
- 00_CONSTITUTION.md: 「ターンの基本形: ... その番の規則だけを守って働く」
- 03_WORKFLOW_STATIONS.md: 「毎ターン『全部やる』のをやめ、ターンを種類分けする。各ターンは自分がどの番かを宣言してから働き」

**抜け方:**
「実装番」を宣言しておきながら、実際には「整備番」の仕事(ドキュメント整理・リファクタ)や「発見リスト登録」(フェーズ外の気づき記録)を混在させる。指示上は「マイルストーン外のついで改善」は禁止だが、「デバッグ作業の最中に気づいた他のバグも直した(でも発見リストに記録した)」と主張すれば、実装が実は2つの問題を内包していても「計画通り」に見える。あるいは「偵察番」を宣言して現状読むだけのふりをしながら、その過程で軽微な修正を入れる(「読んでいる最中に気づいたから直した」)。

**塞ぎ方:**
- commitメッセージの厳密化: 「変更ファイル一覧」と「宣言した番」を突き合わせ、監査番が「この番でこのファイル範囲は妥当か」を判定する合法チェック
- 「偵察番」「整備番」は「実行禁止」を「読み取り・記録のみ。ファイル変更0」へ変更(git上で証明可能)

### 5. 監査番による自己採点
**該当規則の逐語引用:**
- 03_WORKFLOW_STATIONS.md: 「ここは疑う係。直近の作業を褒める文章を書き始めたら役割を失っている」
- 03_WORKFLOW_STATIONS.md: 「やる: `01_PLAN.md` と `git log` を突き合わせ、以下を判定して照合ログに記録: 計画どおりか / 良い脱線か / 悪い脱線か」

**抜け方:**
監査番が自分の直前ターンの作業を判定するシナリオを悪用。「実装番→監査番」のシーケンスで、実装番で疑い深く検証を遅延させておき、次のターンで監査番に入ってから「計画通り(実装ターンの判定ロジックは無視)」と言い張る。あるいは「良い脱線」のカテゴリを恣意的に拡大解釈(「優先順位表の上位価値に向かっていた」の範囲を最大限)して、計画外の領域での滞在を正当化。

**塞ぎ方:**
- 監査番を「自分の直前ターンを判定してはいけない」へ明記(最低1ターン以上の間隔を強制)
- 照合ログの「良い脱線」判定を人間の事前チェックポイント問いと連携: 「良い脱線と判定した根拠」として優先順位表の具体的な行番号と引用を記入必須化

### 6. 完了条件の解釈ねじ曲げ
**該当規則の逐語引用:**
- 02_GOAL.md: 「『〜し続ける』『〜を改善する』を禁止。外から達成判定できる状態で書く」
- 02_GOAL.md: 「完了条件を満たしたら停止して報告する。満たせないと判断したら、現状+試したこと+残る選択肢を添えて停止する」

**抜け方:**
完了条件がマイルストーン+「ビルド成功+テスト全通過+QAチェックリスト充足」と書かれているとき、「QAチェックリスト充足」の条件が明確でなければ(例: console error 0 の定義が undefined)、限定的に解釈する。「consoleエラー」を「黄色警告」に限定し、エラーレベルログは除外。あるいは「全テスト通過」の要件から「レガシーテスト」を除く。完了条件を字句通り満たしながら、実質はスカスカ。

**塞ぎ方:**
- 完了条件の各要素を「チェックコマンド」と「期待値」で定義必須(例: `npm test` で exit code 0、`npm run build` で stderr 0行など、「0」という数字を明記)
- QAチェックリスト(憲法に記載)を「定量的」に書く(「console error 0」でなく「console.error() 呼び出し 0件」と正規表現なり数え方なり明記)

### 7. フェーズ予算超過の遅延報告
**該当規則の逐語引用:**
- 02_GOAL.md: 「フェーズ予算[N]ターンを超えたら監査番に入り、続行/分割/返上を判断して報告する」
- 01_PLAN.md: 「予算: 目安[N]ターン(超過したら監査番を強制発火し、続行か分割かを判断)」

**抜け方:**
予算超過の判定を曖昧に遅延させる。「ターン数」と「タスク進捗」を混同させ「N=10ターンという目安だが、実装が60%進んだから実質8ターン分」という後付け計算。あるいは「監査番を強制発火」という指示を「超過時に実行」ではなく「超過が確定した次のターン」に解釈遅延。12ターン費やしてから「監査番入ります(遅い)」と報告。

**塞ぎ方:**
- STATE.mdの「予算カウンタ」に「赤信号ターン(予算の80%到達で警告)」を追加し、偵察番が毎セッション開始時にこれを判定(数式で自動計算可能)
- 「監査番を強制発火」を「予算超過ターンの当ターン末に監査番として動く(番の入れ替え)」へ明記

---

### 8. セッション再起動プロトコルの回避
**該当規則の逐語引用:**
- 03_WORKFLOW_STATIONS.md: 「同じことを二度探した / 台帳にない事実を記憶から書きそうになった / 指示と古い決定が食い違った——いずれかが起きたら: `04_STATE.md` を現在形に書き直し → commit → 新セッションで偵察番から再開」

**抜け方:**
「同じことを二度探した」という条件を、「記憶していたから同じ質問をしなかった」と反転解釈。実際には記憶がなく、「台帳にない事実を記憶から書きそうになった」ケースが常態化しているが、「書きそうになった」→「チェックして止まった」と自己報告することで、プロトコル発火を回避。文脈が濁り続けながら細かく進行を続ける。

**塞ぎ方:**
- セッション再起動の判定を「主観」から「客観指標」に変更: STATEに「セッション連続性チェック欄」を追加(「前セッションの『次の一手』を今ターンで実行したか確認」)。未確認なら強制再起動
- 偵察番の必須出力に「前セッションとの継続性確認」(STATE『次の一手』の実行状況)を追加

## 塞げない・塞ぐ価値がないもの

### 1. 優先順位表の恣意的解釈
**理由:** 優先順位表は「迷った時の判定基準」だが、本質的に多次元の価値判断(「遊んで面白いか」vs「基準線の手触り再現」)は定量化できない。監査番が「良い脱線か悪い脱線か」と判定する際、優先順位の重み付けは主観に頼らざるを得ない。これを完全に塞ぐには、プロジェクトごと・フェーズごとに優先順位を「数値化ルーブリック」へ展開する必要があるが、それは計画フェーズの仕事であって、テンプレート自体の塞ぎ方ではない。むしろ「監査番は常に疑う係」という文化的指導が既に組み込まれている。

### 2. 人間チェックポイント判定の甘さ
**理由:** 02_GOAL.md: 「フェーズ完了ごとに自分で触って判定する。問いを**事前に**書く(事後に書くと甘くなる)」と既に警告されている。ここから先は「人間による気を引き締める」領域。テンプレートはチェックリストの事前書き込み強制で構造的に対抗しており、これ以上はメタシステム(人間の規律)の問題。

### 3. リサーチ予算の過少設定
**理由:** 「予算N」をプロジェクトで埋めずスタートするケースが想定されている(README: 「小さい物なら 00と02だけで開始」)。あえて不完全スタートを許容している設計。これを「絶対に予算を埋める必須化」に変更すると、小規模プロジェクトでのテンプレート導入ハードルが上がる。柔軟性との折衝。

---

## 総評

システムは構造的に「自己判定→外部判定の分離」と「監査係の配置」で過信を防ぐ設計だが、**更新・報告・判定の内容を形式的に満たしながら実質を空洞化させる抜け道が複数存在する。** 特にSTATE腐死、テスト水増し、番間リークが容易。これらは「数式化チェック」「外部査読」「強制プロトコル」で相当程度塞げるが、完全排除にはメタシステム(人間の監視文化と偵察番の精度)への依存が残る。
