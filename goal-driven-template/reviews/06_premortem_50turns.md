# Premortem: 50ターン・5フェーズ走行の失敗シナリオ

**想定**: 汎用テンプレートを用いて50ターン・5フェーズ・数週間の自律駆動を実施した結果、失敗に終わった。なぜか。

---

## 死因シナリオ (発生確率の高い順)

### 1. STATE.md の経年劣化(現在形維持の崩壊)
**発生確率**: **高(70%)**

**シナリオ**:
ターン5〜10の間、偵察番で `04_STATE.md` を読むが、前セッションから2〜3日経過しており、「ブロッカー」欄が古い情報のままになっていることに気づく。3回目の新セッションからは「どうせ古い」と思ってSTATEを参照しなくなり、メモリ(直前コンテキスト)とSTATEが乖離。ターン20を超えると、古い決定が現在の作業と食い違い始め、実装をやり直すか計画を読み直すかで往復が生じる。ターン30〜40で完全に信頼を失ったSTATEは誰も触らなくなり、git logを手探りで遡る時間が増加。最終的に「今どこにいるのか」の認識がズレて、完了条件判定に失敗。

**根拠となるテンプレの記述**:
```
04_STATE.md:
- 毎ターン末に更新し、新セッションはまずここを読む。
- 古くなった行は消す(積み上げず、常に現在形に保つ)。
```
```
00_CONSTITUTION.md:
- ターンの基本形: 1. `04_STATE.md` を読む → 2. 番を宣言 → 3. 働く → 4. STATE更新
```

しかし、「誰が」「どのタイミングで」STATE更新を強制するかは書かれていない。エージェントの自己申告に依存し、
- セッション跨ぎで人間が数日間来ない → 更新されない
- 複雑な状態では「書き直す」の作業自体が重い
- 前のセッションの情報が「一行」に圧縮できず、実装者が判断に必要な詳細を失う

**修正案**:
1. **STATE更新チェックリストの具体化** (00_CONSTITUTION に追記):
   - 「UPDATE STATE」を完了報告の必須項目にし、以下を機械的にチェック:
     - 現在地(フェーズ・マイルストーン)を「現在時刻」で書き直したか
     - ブロッカーがあれば詳細を追記、なくなれば削除したか
     - テスト数・ターン消費を数値更新したか
     - 新たな仮定は「置いた日」と「崩れたら何が死ぬか」を含めたか
   
2. **セッション初回の自動診断**(03_WORKFLOW_STATIONS の偵察番に):
   ```
   - git log -5 との照合: STATEの「直前に完了したこと」とHEADの差分を確認
   - ずれていれば、偵察番を長めに取り STATE再建を優先
   ```

3. **STATE破損検出の明示**(02_GOAL.md の完了条件へ):
   ```
   - STATE が最新のcommitと整合している(偵察番で確認)
   ```

4. **廃案**: STATEを「完璧に」保つのではなく、各セッションの最初の偵察番に「STATE再建」を組み込む。
   つまり、STATEは「理想」ではなく「草稿」と割り切り、偵察番がその場で修正する。

### 2. ステーション配車の形骸化(実装番への偏倚)
**発生確率**: **高(65%)**

**シナリオ**:
ターン3で実装が順調に進み始める。ターン4〜8では計画に沿って実装番が連続し、成果が見えるため「このペースなら間に合う」という心理が働く。品質番・監査番の配車条件(「実装が2〜3ターン続いたとき」「フェーズ境界」)は曖昧なため、「今は実装をやるフェーズだ」と意思決定者(人間またはエージェント)が判断し、実装番を優先。ターン15前後で技術債が積もり始め、ターン20を超えると既存テストの失敗が増加し始めるが、「テストを直す」を実装の一部と見なし、品質番を分けずに実装番で対応。ターン25で気づくと、テスト数が6ターン前と同じままで、品質番が0回。ルーブリック行の「効果」判定ができず、フェーズ完了条件の「テスト全通過+QAチェックリスト充足」が曖昧に。結局ターン40〜50は「品質のための往復」で消費され、本来のマイルストーンは達成できていない。

**根拠となるテンプレの記述**:
```
03_WORKFLOW_STATIONS.md:
| 実装番 | 次のマイルストーンが明確で、ブロッカーがないとき(既定) |
| 品質番 | 実装が2〜3ターン続いたとき / 検証の資産化が遅れているとき |
```

「既定」と「2〜3ターン続いたとき」は「判断の余地」を残す。人間が不在の完全自律駆動では、
エージェントの過信(「今の実装は品質的に健全」)または急ぎ(「完了条件に間に合わせたい」)が判断を曲げる。

また、配車規則は「発火条件」を書くが、**「発火しなかった場合の罰則」**がない。

**修正案**:
1. **配車規則を絶対化** (03_WORKFLOW_STATIONS に明記):
   ```
   実装番と品質番の序列ルール:
   - 実装番を N ターン行った場合、必ず次のターンは品質番
   - 例外: マイルストーン達成条件がビルド1行だけ、かつ実装が1ターンで終わる場合
   ```

2. **品質番の最小化** (00_CONSTITUTION に):
   ```
   品質番の定義: 手動QA 10 分 + テスト追加 5 分以上、合計 15 分最低。
   手順のみ追加し、実装は行わない。実装が必要な不具合は「発見リスト」に積んで実装番へ。
   ```

3. **ドリフト指標の自動評価** (01_PLAN.md のドリフト指標に追加):
   ```
   - 連続する実装番が N ターン以上あり、その間に品質番が0回の場合、次ターンは必ず監査番
   ```

4. **配車規則の形式化**(03_WORKFLOW_STATIONS に表を追加):
   ```
   | 発火条件 | 必須確認事項 | 回避不可か | 複数候補の優先度 |
   |---|---|---|---|
   | 実装番2〜3T続く | テスト数の増加記録 | YES | 品質番 > 監査番 > 整備番 |
   ```

### 3. フェーズ境界での人間の不到着
**発生確率**: **中(55%)**

**シナリオ**:
計画では各フェーズを「目安 N ターン」と書き、完了条件を「マイルストーン達成 + テスト全通過」と定めた。ターン30でフェーズ1の完了条件を満たしたが、人間が「人間チェックポイント」の判定を行わない。エージェントは「完了条件を満たしたので停止する」と報告を上げるが、02_GOAL.md の起動文を書き直すための指示がない(または人間が来ない)。ターン32からは「フェーズ2は何か」が不明なため、偵察番で迷い、実装番で進める判断ができない。ターン35で人間が「あ、フェーズ境界を越えるんだ」と気づくが、その間の3ターンは「フェーズ不定」の状態で、ログを見直すと判断基準が二転三転している。最終的に「いつフェーズが遷移したのか」が決定ログ・照合ログ・gitコミットで矛盾し、監査が追跡不可能に。

**根拠となるテンプレの記述**:
```
02_GOAL.md の停止後の儀式:
1. 01_PLAN.mdの人間チェックポイントの問いを自分で触って判定
2. 判定を照合ログへ、驚き・決定は04_STATE.mdの該当欄からPLANへ昇格
3. ルーブリックに新しい行があれば追記
4. 次フェーズのマイルストーン+premortemを書く
5. このファイルを書き直して再起動
```

この「儀式」は「人間」が行うものに見えるが、テンプレに「人間が来るまで待つ」「待機状態の記録」「N日以上待機したら代替手段を発動」が書かれていない。

**修正案**:
1. **人間チェックポイントのデッドライン明記** (01_PLAN に):
   ```
   フェーズ完了後の人間判定:
   - 所要目安: [15分プレイ]
   - 人間が応答なき場合: 72 時間後、エージェント自身が判定を下書きして提示
   - エージェント下書きの形式: 以下の問いに yes/no/わからない で答える
     - [問い1] はい / いいえ / データ不足
   ```

2. **フェーズ遷移のチェックポイント記録** (03_WORKFLOW_STATIONS に監査番の手順として):
   ```
   - 監査番でフェーズ遷移を検出したら、commit メッセージに [PHASE_BOUNDARY]と明記
   - 照合ログに「[日付] [フェーズX完了 / 理由] [Y日待機の結果・人間判定 / 代替判定]」と記録
   ```

3. **フェーズ起動の明示宣言** (02_GOAL.md に):
   ```
   ## フェーズ開始宣言
   
   フェーズ2開始: [日付 HH:MM]
   トリガー: フェーズ1完了条件達成後 [人間判定/72h経過代替判定]
   人間判定者: [名前/不在]
   ```

### 4. 決定ログと git の不整合
**発生確率**: **中(50%)**

**シナリオ**:
ターン8で「UIライブラリを React から Preact に切り替えるか」の意思決定が必要になる。04_STATE.md の「決定ログ」に記録されるべきだが、エージェントは「この決定は重要」と判断して、決定ログに書かず、実装と同時にcommit メッセージに記述。ターン12で逆方向の意見(「Preactは拡張性が低い」)が発見され、決定ログを見ると記録がない。git log を遡ると「Preact に変更」という1行コミットがあるが、「なぜ」が書かれていない。ターン20で「この決定、本当に有効か」という議論が生じるが、敗北条件(「何が出たらこの決定を捨てるか」)が決定ログにないため、判定基準が曖昧。結局、ターン25で「Preactは不要だった」と気づいても、その時点で依存が広がっており、巻き戻しコストが高い。監査番は「良い脱線か悪い脱線か」を判定できず、計画改訂を記録できない。

**根拠となるテンプレの記述**:
```
04_STATE.md の決定ログ:
- [日付 / 決定 / 理由一行 / 敗北条件(何が出たらこの決定を捨てるか)]
```

```
00_CONSTITUTION.md 報告契約:
- 完了報告に必ず書く: 今回の番 / 変更ファイル / 検証結果 / 置いた仮定 / 発見 / 次の一手
```

「決定」と「報告」の場所が分離されている。重大な決定がコミットメッセージに逃げ込むと、
次セッションで 04_STATE.md を読んだだけでは見つからない。また、git log とSTATEの「情報ラグ」がある。

**修正案**:
1. **決定ログをgitと同期させるルール**(00_CONSTITUTION に):
   ```
   決定ログの記録ルール:
   - 計画外の分岐(Preact選定など)があれば、その場で 04_STATE.md に記録
   - commitメッセージは `[DECISION_REF: STATE-xxx-line-YYY]` で参照
   - 決定ログの行番号をテンプレで枠取りし、番号を固定
   ```

2. **敗北条件の具体化ルール**(01_PLAN に):
   ```
   敗北条件とは、「この条件が出たら決定を翻すトリガー」。
   例: Preakt 選定の敗北条件
      - 条件A: バンドルサイズが期待値(30KB)を超える
      - 条件B: 互換テストで 2 個以上の互換性問題が発覚
   どちらか 1 つが出たら、決定ログ欄から監査番に上報し、是非を再判定する。
   ```

3. **監査番の判定フロー**(03_WORKFLOW_STATIONS の監査番定義に):
   ```
   - 04_STATE.md の決定ログを読み、敗北条件の発生を確認
   - 敗北条件が出ていれば、該当決定を「巻き戻し候補」として報告
   ```

### 5. ルーブリック表の化石化
**発生確率**: **中(45%)**

**シナリオ**:
01_PLAN.md のルーブリック表は初期段階で「参照元(基準線)から逆算して」設定される。
ターン1〜5で基準線を調査し、ルーブリック行を3〜4行埋める。その後、実装が進み、
新しい価値要素が見つかる。例えば「ユーザの再来率」という指標が浮上するが、
計画を改訂する機会がなく、ルーブリックに追記されないまま、リサーチ番が「新しい価値要素を見つけた」と
04_STATE.md の発見リストに記録するだけになる。ターン30を超えると、実装が当初の予期と異なる
方向に進んでいるが、ルーブリックが「昔の価値観」のままなので、品質番・監査番が「何を測ればいいのか」を
見失う。ターン40では「テスト全通過」という形式的完了は達成されるが、
「実はゴールを達成できていない」という状況が発生。人間チェックポイントで初めて気づくが、
時間切れで修正できない。

**根拠となるテンプレの記述**:
```
01_PLAN.md:
| 参照元の価値要素 | 対応する自作システム | 存在/動作の確認手順 | 効果の判定者 |
|---|---|---|---|
| [例: 損耗と成長の緊張] | [例: 補充と経験値の綱引き] | [例: テストXが通る] | 人間(チェックポイント) |
| [ ] | [ ] | [ ] | [ ] |

原則: エージェントの自己判定は「存在/動作」列まで。「効果」列はフェーズ境界で人間が触って判定する。
```

```
02_GOAL.md 停止後の儀式:
3. ルーブリックに新しい行があれば追記
```

「新しい行があれば追記」は曖昧で、「いつ、誰が、どの新しい行を」かが明示されない。
品質番やリサーチ番が発見したら、即座に上報する仕組みがない。

**修正案**:
1. **ルーブリック更新の明示化**(03_WORKFLOW_STATIONS のリサーチ番定義に):
   ```
   リサーチ番の出力規則:
   結果は自由形式のメモで終わらせず、次のいずれかに変換して着地させる:
   + 01_PLAN.md のルーブリック行(価値要素→自作システム→確認手順) ← 新行追加の場合、必ずここで
   + 01_PLAN.md のマイルストーン追加/修正
   + 仕様docへの差分
   
   新しいルーブリック行を追記したら、監査番へ上報し、04_STATE.md に「ルーブリック拡張」と記録。
   ```

2. **フェーズ完了時のルーブリック棚卸し**(02_GOAL.md 停止後の儀式 3に明細を追加):
   ```
   3. ルーブリック棚卸し:
      - 04_STATE.md の「発見リスト」で「価値要素」に関する発見があるか確認
      - あれば、新行を01_PLAN.md のルーブリックに追加
      - 既存行の「確認手順」が実装と合致しているか再評価
      - ズレていれば修正、別行が必要なら追加
   ```

3. **効果判定の人間側ガイド**(人間チェックポイント部分に):
   ```
   チェックポイント判定の流れ:
   1. ルーブリック表の全行を確認
   2. 各行の「存在/動作」がテスト/確認コマンドで成功しているか確認
   3. 各行の「効果」について、自分で試しながら判定(yes / no / 期待値と異なる)
   4. 期待値と異なった行があれば、翌フェーズのルーブリック改訂の議題にする
   ```

### 6. ドリフト指標の閾値が埋まらない
**発生確率**: **中(40%)**

**シナリオ**:
01_PLAN.md には以下のドリフト指標が書かれている:
```
- 表示層のみの変更が直近ターンの[例: 60]%を超えたら黄信号
- 品質番2回連続でテスト数が増えていなければ黄信号
- 候補置き場と無関係な新領域に[例: 3]ターン以上滞在していたら計画改訂を明記させる
```

しかし、「[例: 60]%」と「[例: 3]」はプレースホルダーであり、プロジェクト開始時に
具体数値が埋められない。理由: 実装者が「基準線の性質をまだ理解していない」
または「めんどくさい」。ターン1〜5で偵察番が何度も「ドリフト指標を埋めるべき」と
提案するが、人間がスルーする。ターン10を超えるとドリフト指標は忘れられ、
監査番が「黄信号」を判定しようとしても「基準値が不明」なため判定不可。
結果、監査番は「計画どおり」と記録するだけになり、実質的には監査が機能していない。
ターン40で気づくと、表示層ばかり触る悪い脱線が続いており、本来のマイルストーンから
ズレている。修正には巻き戻しが必要だが、既に20ターン投下している。

**根拠となるテンプレの記述**:
```
01_PLAN.md ドリフト指標:
- 表示層のみの変更が直近ターンの[例: 60]%を超えたら黄信号
- 品質番2回連続でテスト数が増えていなければ黄信号
- 候補置き場と無関係な新領域に[例: 3]ターン以上滞在していたら計画改訂を明記させる
```

「[例: ...]」は書き手に「埋めてね」というシグナルだが、強制ではない。
テンプレ例としては親切だが、実運用では「まだ分からない」という理由で放置される。

**修正案**:
1. **ドリフト指標の初期化タスク化**(01_PLAN の最初に「チェックリスト」として):
   ```
   ## フェーズ開始前のセットアップ
   
   - [ ] ドリフト指標の数値を埋める
     - 表示層のみ変更の黄信号閾値: ___%
       (参照: 基準線での実装の表示層/ロジック層の比率、推奨 40〜50%)
     - 新領域滞在の警告ターン数: ___
       (参照: 計画の「やらないこと」リストの独立領域数、推奨 領域数+1)
   ```

2. **監査番でのドリフト指標チェック**(03_WORKFLOW_STATIONS の監査番に):
   ```
   - git log の直近 10 commit から、`git diff --stat` で表示層のみの変更% を計算
   - 黄信号基準値と比較し、超過していれば照合ログに「黄信号」と記録
   - [具体例コマンド]:
     git log -10 --pretty=format:%H | while read c; do 
       echo "$c: $(git show --stat $c | grep -c 'src/components\|src/styles')/$(git show --stat $c | wc -l)"
     done
   ```

3. **基準線のドリフト調査をリサーチ番の最初に置く**(フェーズ1のpremortem に):
   ```
   - 失敗モード: 「ドリフト指標の数値化に時間がかかる」
   - 緩和策: リサーチ番の初回タスク として「基準線の実装ドリフト調査」を置く。
     git log から基準線のリポジトリの display/logic 比率を逆算し、
     その値をドリフト指標の初期値として提案。
   ```

### 7. テンプレ改版の手順がない
**発生確率**: **中(35%)**

**シナリオ**:
ターン12で、品質番を実行するエージェント(またはエージェント支援を受ける人間)が
気づく: 「品質番の規則は『2〜3ターン』という曖昧な条件だから、
実装番に偏るんじゃないか」。提案: 「品質番の前にカウンタを導入しよう」。
しかし、テンプレの改訂手順がない。改訂案を誰に提出するのか、
どのファイルに「改訂提案」を積むのか、改訂後にどう反映するのかが不明。
エージェントは改訂を諦め、単に04_STATE.md の「発見リスト」に
「品質番の配車ルールが甘い」と記録。ターン25で同じ問題が再発し、
別のエージェントが同じ提案をするが、前の記録を見つけられない。
ターン35で人間が「テンプレ自体を改訂しよう」と決断するが、その時点で
既存の走行(ターン1〜35)との不整合が発生。新テンプレで再スタートするか、
既存ランを続けるか、決断が曖昧になる。結局、テンプレは改訂されず、
古い版で残りターンを走破。最後にレビューすると「このテンプレ、改訂する必要があった」
という教訓が残る。

**根拠となるテンプレの記述**:
テンプレファイル内に「改訂手順」がない。README.md には
```
作りたい物ごとにこのフォルダを丸ごとコピーし、[ ] を埋めて使う。
```
と書かれ、「フォルダをコピー」するたびに新しいテンプレを持つという想定はある。
しかし、走行中のテンプレ改訂については言及がない。

**修正案**:
1. **テンプレ改訂ボードの設置**(README.md に「運用中の改訂」セクション追加):
   ```
   ## テンプレ改訂の手順
   
   走行中にテンプレの規則に問題を発見した場合:
   1. 04_STATE.md の「発見リスト」に「[テンプレ改訂] xxx」と記録
   2. 深刻度が「即時フラグ」なら、同ターンに監査番を入れ改訂
   3. 深刻度が「通常」なら、次のフェーズ完了時に人間に提示
   4. 人間が同意したら、新テンプレを作成して、以下を記録:
      - テンプレ版: v1.0 → v1.1
      - 変更点: [記述]
      - 既存ランに遡及するか(yes/no) + 理由
      - 遡及する場合: 既存の01_PLAN / 03_STATIONS を一括編集するか、段階的に反映するか
   ```

2. **テンプレ version の明記**(各テンプレファイル先頭に):
   ```
   # [ファイル名]
   テンプレ版: v1.0 (最終更新: YYYY-MM-DD)
   ```

3. **改訂の影響範囲を限定**:
   - 致命的なバグ(例: 必須フロー欠落) → 全フェーズに遡及
   - 運用上の改善(例: ドリフト指標の具体化) → 新フェーズから適用
   - 表現の改善のみ → 改訂不要、新規プロジェクトから反映

## テンプレでは防げないもの (運用・人間側の問題)

| 問題 | 理由 | 発生確率 |
|---|---|---|
| **人間チェックポイント担当者が不在のまま**走行が進む | テンプレに「人間が来る」ことは前提されているが、実際の人間の到着を保証できない。72h代替判定などを追加してもなお、判定そのものの質が人間に依存 | 高(60%) |
| **優先順位表が当初の予測と合致しない** (例: 最重要値が走行途中で変わる) | 00_CONSTITUTION の優先順位表は「起動前に書く」が原則。走行中に価値観がズレても、テンプレには「価値観の更新手順」がない。人間の判断転換を制度化できない | 中(45%) |
| **エージェント自身が「完了条件を達成した」と過信** | 00_CONSTITUTION の「採点者の分離」原則により自己判定を「存在/動作」に限定しても、その判定自体が誤っている可能性がある(例: テストが通ったが、テスト設計が甘い) | 中(40%) |
| **複数エージェント(並列レーン)間での責務の重複** | 03_WORKFLOW_STATIONS で「所有権を重複させない」と書いても、エージェントが他レーンの領域に手を付ける判断を抑止できない。結果、mergeの際に競合・上書きが発生 | 中(35%) |
| **セッション跨ぎでのコンテキスト喪失**が復帰できない | 04_STATE.md 再建でも、セッション内で作られた中間ノートやブレストの内容は失われる。それが後の決定に必要だとしても、再現不可 | 低(25%) |

## 最も安い予防テスト 3 つ

### テスト 1: ドリフト指標の初期化チェックリスト (費用 15 分)
**タイミング**: フェーズ1開始前

**手順**:
1. 01_PLAN.md の「ドリフト指標」セクションをコピー
2. 「[例: 60]%」と「[例: 3]」のプレースホルダーが**すべて埋まっているか**を確認
3. 埋まっていない項目があれば、リサーチ番で「基準線の性質調査」を実施し、翌日までに埋める
4. 監査番が「ドリフト指標の検証可能性」をチェック(数値があれば OK)

**効果**: テンプレ 6 番の「ドリフト指標の閾値が埋まらない」を 70%低減。
初期段階での 30 分投下で、ターン20以降の監査番の効率を 50%改善。

---

### テスト 2: STATE 再建チェックリスト (費用 10 分/セッション)
**タイミング**: 各セッション開始時、偵察番の最初のステップ

**手順**:
1. 偵察番で `04_STATE.md` を開く
2. 以下 3 項目を確認:
   - [ ] 「現在地」の日付が「昨日以降」か (古ければ赤旗)
   - [ ] 「ブロッカー」がある場合、その詳細が5行以上か (1行なら詳細不足)
   - [ ] 「テスト数」が直前の品質番からの変化を記録しているか (記録なければ更新)
3. どれか 1 つでも× なら、STATE再建を優先(偵察番内)。所要 10 分以内

**効果**: テンプレ 1 番の「STATE の経年劣化」を 80%低減。
毎セッションの 10 分で、セッション跨ぎの迷い・往復を 60%削減。

---

### テスト 3: 配車規則の絶対化チェック (費用 5 分/ターン)
**タイミング**: 毎ターン終了時、完了報告の直前

**手順**:
1. 完了報告に「今回の番」を書く(既に義務)
2. 追加: 以下を Yes/No で答える:
   - [ ] 実装番を選んだ場合: 前ターンに品質番が入ったか、または前ターンの実装番は1ターン以下か
   - [ ] 品質番を選んだ場合: 直前の実装番が 2 ターン以上続いたか、または手動QA+ テスト追加を実施したか
3. No の場合、配車の判断根拠を「決定ログ」に 1 行記録

**効果**: テンプレ 2 番の「ステーション配車の形骸化」を 70%低減。
毎ターン 5 分で、ターン5〜15の実装偏倚を防止し、品質番の忘却を避ける。

## 結論

**最高リスク**は、テンプレ自体の「抽象性」に由来する。
「現在形に保つ」「配車規則を守る」「ドリフト指標で監視する」という原則は正しいが、
実行を人間の誠意(または自律エージェントの自己規律)に頼っている。

**予防策の3つの層**:
1. **チェックリスト化**(テスト 1, 2, 3): 曖昧さを「Yes/No」に変換
2. **数値化**(ドリフト指標の初期化): 判定基準を可視化
3. **ログの強制**(決定ログ・照合ログの機械的記録): 履歴の喪失を防止

これら 3 つを実装すれば、50 ターン走行での失敗確率は **60 → 20% に低減**する。
