# 汎用テンプレート汎用性レビュー

**対象:** 6ファイル(README.md, 00_CONSTITUTION.md, 01_PLAN.md, 02_GOAL.md, 03_WORKFLOW_STATIONS.md, 04_STATE.md)

**視点:** ゲーム開発由来の骨格を、3つの異質なドメイン(技術書執筆、データ分析、API保守)で検査。各ケースで埋められない[ ]、意味を失う規則・ステーション、ゲーム前提の漏れた語彙、欠ける工程を特定。

---

## ケース別の破綻箇所

### ケース1: 技術書の執筆(コードがない・ビルドがない・目に見える「面白さ」がない)

#### 埋められない[ ]
- **00_CONSTITUTION.md § 定数**
  - `ビルド: [コマンド]` — ビルド自体が存在しない
  - `テスト: [コマンド]` — テストケースが定義できない(著者の主観的判断が最終)
  - `QA環境: [例: desktop 1440x900]` — 意味をなさない
  - `QAチェックリスト: [例: console error 0 / broken image 0 ...]` — 技術的エラーがない

#### 意味を失う規則・ステーション
- **03_WORKFLOW_STATIONS.md § 実装番**
  - 「製品版構造に薄く通す」の意図は理解可能だが、「**ビルド成功**+既存テスト全通過。失敗したまま次に進まない」は実行不可
  - 完了条件が「章の完成度が基準線(参考書)に見劣りしないか」という主観的判定に変わるが、テストで検証できない

- **03_WORKFLOW_STATIONS.md § 品質番**
  - 「手動QAをテストに変換して保存」が破綻(書籍はテスト対象ではない)
  - テスト数が「フェーズを通じて単調増加が正常」とする前提が成立しない

#### ゲーム前提の漏れた語彙
- **01_PLAN.md § ルーブリック**
  - 「対応する自作システム」は、著者の場合「説明の清晰さ」「論理の連続性」など無形要素
  - テンプレートは「システム」「確認手順」という実装物の言葉で書かれており、概念作品では自然な訳語がない

- **04_STATE.md § 予算カウンタ**
  - 「テスト数: [n](品質番ごとに更新。2回連続非増加は異常)」——著者のターンには適用できない指標

#### 欠ける工程
- **査読サイクル** — テンプレートには人間による形式的な「査読」が組み込まれていない
  - 人間チェックポイント(01_PLAN.md)は「効果」を手触りで判定するが、書籍では出版前査読が必須プロセス
  - 「見つからず」がリサーチの成果ではなく「査読で不明確」という外的フラッグが欠けている

- **引用・根拠の追跡** — リサーチ番の「基準線の一次情報を優先」は参考文献リストになっていない
  - 技術書では逆に「ここはどの文献に基づいているか」が著者責務だが、テンプレートに引用管理層がない

### ケース2: データ分析パイプライン構築(UIがない・「面白さ」の即時フィードバックがない)

#### 埋められない[ ]
- **00_CONSTITUTION.md § 定数**
  - `QA環境: [desktop 1440x900]` — パイプラインに「環境」という概念がない(Pythonスクリプト・SQLで動作)
  - `QAチェックリスト: [console error 0 / broken image 0 / overflow なし ...]` — UI QAが不適切
  - 「責務マップ: [パス] / 仕様doc: [パス] / リサーチ置場: [パス]」 — パイプラインでは責務が「データセット名」「変換関数」「検証ルール」であってパスの概念が異なる

#### 意味を失う規則・ステーション
- **03_WORKFLOW_STATIONS.md § コンテンツ番(アセット・データ投入)**
  - 「対象領域のロジックとUI寸法が固まっていること」はゲーム的概念
  - パイプラインでは逆に「ロジックが固まる ≠ データ投入準備完了」(データ品質検査・スキーマ進化・バッチサイズ試行が並走する)
  - 「パイプライン: バックログdoc管理 → 生成元 → 加工済 → 使用版 + manifest」は実装細節にすぎず、ルール化する値がない

- **03_WORKFLOW_STATIONS.md § 偵察番**
  - 「`04_STATE.md` → `01_PLAN.md` の現フェーズ → 責務マップ → git log」の順序は、UIが無い場合に読む痕跡(git log)に比重が高すぎる
  - パイプラインではリサーチ番で「データスキーマ定義」「外部APIの挙動確認」が頻出するが、「責務マップ」の更新頻度がゲーム開発より低い

#### ゲーム前提の漏れた語彙
- **01_PLAN.md § ルーブリック**
  - 「参照元の価値要素 → 対応する自作システム → 存在/動作の確認手順 → 効果の判定者」で、「効果」は「精度」「遅延」「コスト」などの定量指標だが、「人間(チェックポイント)」という判定者は曖昧
  - パイプラインの場合「効果の判定者」は「データサイエンティスト(業務要件照合)」「ML Ops(スケーラビリティ確認)」と分散するが、テンプレートは単一の「人間」を想定

- **02_GOAL.md § 共通雛形**
  - 「ビルドコマンドが成功」はパイプラインでは意味をなすが、「テスト全通過」の定義が曖昧
  - パイプラインテストは「ユニットテスト」「統合テスト」「リグレッション検査」「データ品質チェック」に分化するが、テンプレートは「テスト」を単数名詞で扱う

#### 欠ける工程
- **データ品質の検証ステーション** — コード品質は品質番で捕捉されるが、「データが正しいか」の確認ステーションがない
  - リサーチ番で「外部データソースの挙動」は調査できるが、「投入したデータの品質」を毎ターン検証する仕組みがない
  - 04_STATE.mdに「異常データの発見」を積むことはできるが、構造化された検査はない

- **スケーラビリティ・パフォーマンステスト** — 開発では手元で通っても、本番スケールでは失敗する
  - 品質番に「小規模データセットでのテスト」しか明記されていない

### ケース3: 既存WebサービスのAPI保守(ゼロから作らない・回帰が最重要)

#### 埋められない[ ]
- **01_PLAN.md § フェーズ一覧**
  - 「目的: [一行]」「予算: 目安[N]ターン」——バグ修正や小規模改善では「フェーズ」という概念自体が不自然
  - API保守の仕事: バグ報告 → 最小限の修正 → 回帰テスト → デプロイ ——単一フェーズのループが多く、「フェーズ1 / フェーズ2 / …」に分割する理由がない

- **01_PLAN.md § ドリフト指標**
  - 「表示層のみの変更が直近ターンの60%を超えたら黄信号」——APIではバックエンド限定
  - 「品質番2回連続でテスト数が増えていなければ黄信号」——バグ修正では既存テストの検証が優先であって、新テスト追加が目標でない場合が多い

#### 意味を失う規則・ステーション
- **03_WORKFLOW_STATIONS.md § 実装番**
  - 「製品版構造に薄く通す」——既存APIの場合、構造はすでに固いため「薄く通す」という仮説的試行の前提が異なる
  - 「現マイルストーンの縦切り1つ」——APIのバグ修正では複数の独立したバグ同時対応が常なのに対して、テンプレートは「1つの認識可能な機能」を前提としている

- **03_WORKFLOW_STATIONS.md § コンテンツ番**
  - 「対象領域のロジックとUI寸法が固まっていること」——APIにUIがないため名詞が意味をなさない
  - アセット投入という概念自体が API 保守に無い

#### ゲーム前提の漏れた語彙
- **01_PLAN.md § ルーブリック**
  - 「参照元の価値要素」が「既存のAPIの挙動」を指しているはずだが、テンプレートのサンプル「損耗と成長の緊張」「補充と経験値の綱引き」はゲーム固有
  - API保守では「参照元」= リリース直後の既知の正常状態、または公開仕様ドキュメント。テンプレートはこの「基準線としての過去」を軽視

- **04_STATE.md § 予算カウンタ**
  - 「このフェーズの消費ターン」——バグチケット駆動では「フェーズ」が存在しないため、カウンターが「ターン」から「チケット」に変わる
  - 「テスト数」の単調増加は回帰テスト構築では不自然(テストケースを追加する必要がない既知回帰もある)

#### 欠ける工程
- **回帰テスト・CI/CD検証** — API保守では「本当に既存機能が壊れていないか」の確認が最優先だが、テンプレートは新機能開発のターン順序を想定
  - 品質番にCI通過が含まれるか不明確
  - 「デプロイ前チェックリスト」(監視アラート確認・キャナリア展開確認など)が構造化されていない

- **インシデント対応ステーション** — バグ深刻度に応じた優先順位付けと、本当に深刻なバグの「滞在時間」ルールがない
  - テンプレートの「同一の失敗3回で停止」は予測困難なバグには合うが、本当に重大なバグは1回でエスカレーション必要なことが多い

## 共通する構造的問題

### 1. 「効果」の判定者が明示的でない

テンプレート全体を通じて、「効果」の判定を「人間(チェックポイント)」と抽象化し、判定者を固定せず。

**破綻の形:**
- 技術書: 査読者(編集者・技術レビュアー)の専門分野によって「面白さ」判定がぶれる
- データ分析: データサイエンティストと ML Ops の判定基準が異なる
- API保守: プロダクトマネージャーと SRE の許容度が異なる

**問題:** 「効果」の判定基準が曖昧だと、フェーズ終了判定が恣意的になる。特にゼロから作るゲーム開発では「面白いか」という単一軸で判定できるが、既存プロダクト保守では軸が複数ある。

### 2. ビルド・テストを「外的事実」と過信

テンプレートは以下を前提:
- ビルド成功 = 構造的健全性
- テスト全通過 = 品質確認
- QAチェックリスト = 本番準備完了

**破綻の形:**
- 技術書: ビルド・テストが存在しない。代わりに「章の理解可能性」「論理矛盾の有無」という検査があるが、自動化不可
- データ分析: 複数の検証が並走(ユニット・統合・リグレッション・データ品質・パフォーマンス)し、「テスト全通過」という単一閾値で判定できない
- API保守: 既存テストスイートが大規模であり、「テスト全通過」は満たす必要条件だが十分条件ではない。回帰テストの動的追加がある

### 3. 「マイルストーン」が「可視的な達成」を前提

01_PLAN.md の構造:
```
マイルストーン(各行 = 受入条件 + 検証コマンド):
  - [ ] [達成状態を観測可能に書く] — 検証: [コマンド or 自動テスト名]
```

**破綻の形:**
- 技術書: 「第3章の執筆完了」「査読フィードバック適用」は書けるが、「検証: [コマンド]」の代わりに「査読完了者: [署名]」になり、テンプレート構造と不適合
- データ分析: 「パイプライン完成」は書けるが、本番稼働までのステップ(スケーラビリティ確認・本番環境デプロイ・監視設定)が「マイルストーン」として明確にならない
- API保守: 「修正実装」「リグレッションテスト」「デプロイ」は書けるが、「複数バグの並列処理」では何をマイルストーンにするか不明確

### 4. ステーション定義が「開発ターン」を想定

03_WORKFLOW_STATIONS.md は毎ターン「偵察/リサーチ/実装/品質/整備/監査/コンテンツ」のいずれかを選択する形式。

**破綻の形:**
- 技術書: 「コンテンツ番」(アセット投入)が不適切。著者には「章の外観調整」くらいだが、テンプレートは「ロジックと寸法が固まった領域にアセットを入れる」という開発工程を想定
- データ分析: 「実装番」がコード書きを意図するが、多くのターンは「外部APIから新ビット構造のデータ取得を試す」という試行錯誤であり、コミット対象がない
- API保守: 「実装番」が「バグ修正の実装」に狭くなる。実際には「本番アラートから原因特定」「応急対応と恒久修正の判断」など、テンプレートにない意思決定ターンが多い

### 5. フェーズ境界が「新機能ブロック」を想定

テンプレートでは「フェーズ1で〇〇実装 → フェーズ2で△△追加」という線引き。

**破綻の形:**
- 技術書: 「フェーズ1: 第1章、フェーズ2: 第2章」とは書けるが、「章の間で基礎概念が重複する」など、フェーズの独立性が低い。また査読フィードバックで複数章の修正が必要になる場合、「フェーズ戻り」か「新フェーズ」かの判定が曖昧
- データ分析: 「フェーズ1: ETL、フェーズ2: 分析」と分割できるが、本番デプロイ後の「監視→スキーマ進化→テスト強化」が継続フェーズになるのか新フェーズになるのか不明確
- API保守: 「フェーズ」という概念自体が不自然。バグチケット駆動なので「バグAの修正」「バグBの修正」が sequential に行われ、フェーズ統合の契機が不明確

## 修正案(そのまま適用できる形で)

### 修正案 1: 「判定者」を明示的にマッピングする

**現在:**
```markdown
| 参照元の価値要素 | 対応する自作システム | 存在/動作の確認手順 | 効果の判定者 |
|---|---|---|---|
| [ ] | [ ] | [ ] | 人間(チェックポイント) |
```

**修正:**
```markdown
| 参照元の価値要素 | 対応する自作システム | 存在/動作の確認手順 | 効果の判定者(役割) | 判定基準 |
|---|---|---|---|---|
| [ ] | [ ] | [ ] | [役割: 例: PM / 編集者 / SRE] | [定量 or 定性、具体例] |
```

**例:**
- ゲーム開発: 「面白さ」→ 「プレイ時のテンポ感」→ 「〜秒以内に決定できるか」→ ユーザーテスト参加者 → 定性(サマリー)
- 技術書: 「説明の明瞭さ」→ 「初心者が2回読んでコード実行できるか」→ 査読者が実行テスト → 編集者 + 技術査読者 → 定性(チェックリスト)
- データ分析: 「精度」→ 「予測誤差 < 5%」→ テスト環境での検証 → データサイエンティスト → 定量(スコア)
- API保守: 「安定性」→ 「72時間ノーエラー」→ 本番ログから確認 → SRE + PM → 定量(メトリクス)

### 修正案 2: ビルド・テストを「プロジェクト型」と「保守型」に分ける

**00_CONSTITUTION.md に新セクション追加:**

```markdown
## プロジェクト型の分類(以下から選択)

- **ビルド・テスト駆動** [既定: ゲーム・アプリケーション開発]
  - ビルド: `[コマンド]` / テスト: `[コマンド]` / QA環境: [記述]
  - 実装番の完了条件: ビルド成功 + テスト全通過
  
- **検証駆動(技術文書・コンテンツ)** 
  - ビルド相当: `[例: markdown lint / spellcheck]`
  - テスト相当: `[例: 査読チェックリスト]`
  - 検証: [人力確認プロセス]
  
- **データ品質駆動(分析・ETL)**
  - ビルド相当: `[例: スキーマ検証]`
  - テスト相当: `[例: ユニット + 統合 + リグレッション + データ品質テスト]`
  - 検証: [複数軸の確認]

- **回帰検証駆動(既存プロダクト保守)**
  - ビルド: `[コマンド]` / 回帰テスト: `[コマンド or スイート名]`
  - 本番チェック: [デプロイ前リスト]
  - 完了条件: 既存テスト全通過 + デプロイ前リスト確認
```

### 修正案 3: ステーション定義を「プロジェクト型別」に調整可能にする

**03_WORKFLOW_STATIONS.md に新規セクション追加:**

```markdown
## プロジェクト型別のステーション構成

### 型A(ビルド・テスト駆動: ゲーム・アプリケーション)
[現在のステーション定義をそのまま使用]

### 型B(検証駆動: 技術文書・コンテンツ)
| 番 | 発火条件 | やること |
|---|---|---|
| 偵察番 | セッション開始時 / フェーズ切替 | [現在と同じ] |
| リサーチ番 | 参考文献が不明確 / 事実確認が必要 | 基準線・参考資料を調査 → 仕様doc or 引用マップに記録 |
| 執筆番 | 現フェーズの執筆対象が明確 | 章・節を執筆 → commit(章タイトル) + STATE更新 |
| 査読准備番 | 執筆セッションが2ターン以上 | 論理矛盾・不明瞭箇所をセルフチェック → 査読者リストに送付 |
| 査読反映番 | 査読コメント到着 | フィードバック適用 → commit(対応欄) + 変更理由をSTATEに記録 |
| 整備番 | docs・スタイルガイド遵守が遅れ | 重複削除・用語統一・形式統一 |
| 監査番 | フェーズ境界 / 計画と進捗ズレ | [現在と同じ] |

### 型C(データ品質駆動: 分析・ETL)
| 番 | 発火条件 | やること |
|---|---|---|
| 偵察番 | セッション開始時 / フェーズ切替 | [現在と同じ] |
| リサーチ番 | 外部データソース・API仕様が不明 / スキーマ決定待ち | 基準線(ドキュメント・実データ)を調査 → 仕様docに記録 |
| 実装番 | マイルストーンが明確 / ブロッカーなし | パイプライン実装 → commit + 既存テスト全通過 |
| データ品質番 | 実装が2〜3ターン / 新データソース取込後 | 手動QA(異常検知・スキーマ検査) → テスト追加 + 本番相当スケール試行 |
| パフォーマンス番 | 本番環境導入前 / 処理時間増加兆候 | プロファイリング・負荷テスト → 計画改訂 |
| 整備番 | tech debt が溜まった | リファクタ・ドキュメント整理 |
| 監査番 | フェーズ境界 / 計画と進捗ズレ | [現在と同じ] |

### 型D(回帰検証駆動: 既存プロダクト保守)
| 番 | 発火条件 | やること |
|---|---|---|
| 偵察番 | セッション開始時 / 新バグ受取 | STATE → 担当中チケット → 本番アラート確認 の順に読む |
| インシデント対応番 | 本番障害検知 / 深刻度HIGH以上 | 原因特定 → 応急対応 → commit(インシデントID) + STATE記録 |
| リサーチ番 | バグが再現困難 / ドキュメント不足 | 既知の正常状態(リリース直後の動き)を参照 → 仕様doc再確認 |
| 修正実装番 | バグ原因が特定 / テスト環境で再現 | 最小限の修正 → 既存テスト全通過 確認 |

| デプロイ准備番 | 本番デプロイ前チェック | デプロイ前リスト(アラート確認・キャナリア計画・ロールバック策)確認 |
| 整備番 | 複数チケット対応後 / テスト負債 | テスト整理・ドキュメント更新 |
| 監査番 | 定期(週1回など) / MTTRが目安超過 | チケット進捗と優先度ズレを検査 → 計画改訂(ホットリスト化など) |
```

```markdown
## フェーズ定義(プロジェクト型別)

### 型A(ビルド・テスト駆動)
フェーズ = 独立した機能ブロック + 独立した価値提供。
例: フェーズ1 ゲームロジック実装 / フェーズ2 UI構築 / フェーズ3 チューニング

### 型B(検証駆動: 文書)
フェーズ = セクション + 外部査読サイクル(編集者 + 技術査読者)。
例: フェーズ1 第1〜3章執筆 + 査読反映 / フェーズ2 第4〜5章 + 通読査読

### 型C(データ品質駆動)
フェーズ = パイプラインの段階 + 本番検証。
例: フェーズ1 ETL実装 / フェーズ2 本番検証・最適化 / フェーズ3 運用態勢整備

### 型D(回帰検証駆動)
フェーズ概念を使わない。かわりに **「チケット駆動」** を採用:
- ホットリスト(本番障害):  単一チケット = 最短ターンで対応 → 定期監査で優先度確認
- 通常バグ: 複数チケットを並列処理 → 週1回の監査番で優先順位再確認
- 改善要望: 累積して四半期ごとに「小フェーズ」に昇格

フェーズを使う場合は「四半期ごとの改善フェーズ」程度に限定し、
日々の保守作業は ステーション制で回す。
```

```markdown
このテンプレートは4つの型に対応しています。以下から選んでください。
```

---

## 汎用化しない方がよい部分(理由付き)

### 1. 「ステーション制」の廃止提案は却下

**理由:** 毎ターン「今何をするか」を宣言させることで、エージェントが「方針を持たずに全部やろうとする」失敗を防止できる。これは4つの型すべてで有効。むしろ型ごとにステーション定義を変えるべき。

### 2. 「採点者の分離」(効果は人間の領分)を弱めない

**理由:** エージェント自己採点の過信を防ぐこの原則は、型A・C では特に重要。型B・Dでも「判定者」を明示化することで原則は生きる。曖昧にしてはいけない。

### 3. git commit ターンごとという習慣を変えない

**理由:** 4つの型すべてで「何が変わったか」を記録する必要がある。

- 型A: コード行数・テスト数の客観指標
- 型B: 執筆行数・査読回答数
- 型C: テストケース数・データセット更新
- 型D: バグID・パッチ番号

むしろ commit メッセージ形式を型別に定めるべき。

### 4. 「基準線」(参照元)の概念は変えない

**理由:** 「何が良いか」を定義する外部アンカーは、型A ゲーム(参照作品)、型B(参考文献・スタイルガイド)、型C(ベンチマーク)、型D(既知の正常状態)と変わるが、概念そのものは消えない。

型ごとに「基準線の取得方法」を明示するだけ。

### 5. premortem を廃止しない

**理由:** 「このフェーズは失敗した。なぜか」という逆算思考は、4つの型すべてで有効。失敗モードは型ごとに変わるが。

例:
- 型A: 「ビルド環境の不備」「スコープ増加」
- 型B: 「査読者のスケジュール」「論理矛盾の見落とし」
- 型C: 「外部APIの仕様変更」「データ異常の検知遅れ」
- 型D: 「デプロイ環境と開発環境の乖離」「複数チケットの優先度判定ミス」

## 最終評価: 汎用性の度合い

### 骨格は汎用的 → 型別の調整で吸収可能

| 要素 | 汎用性 | 調整の必要度 |
|---|---|---|
| 層の分離(憲法/計画/ゴール/ステーション/状態) | ★★★★★ | なし |
| ステーション制 | ★★★☆☆ | **高**(型別定義) |
| 採点者の分離 | ★★★★☆ | **中**(判定者マッピング) |
| 記録層の分担(git/STATE/docs) | ★★★★★ | なし |
| フェーズの概念 | ★★★☆☆ | **高**(型D は廃止 or 軽量化) |

### 実装上の提言

1. **README に型選択フロー追加** — ユーザーが最初に型を選ぶことで、以後の記入ミスを減らせる

2. **00_CONSTITUTION.md を型別セクションに分割** — 混ぜ込まずにセクション化し、非該当部分は削除指示を明記

3. **01_PLAN.md のテンプレートを型別に多重化** — 現在の雛形を型A と呼び、型B/C/D の雛形を並列提供

4. **03_WORKFLOW_STATIONS.md を型別に表分割** — 配車規則の表を型ごとに用意し、「使う行」を明示

5. **修正案実装時の優先度:**
   - **P0(必須)**: 修正案1(判定者明示) + 修正案8(型選択) → テンプレート本体に組込
   - **P1(推奨)**: 修正案2/3(ビルド・ステーション型別) + 修正案4/5(マイルストーン・フェーズ柔軟化)

## 結論

**汎用テンプレートとしての成立度: 7/10**

**成功している点:**
- 層の分離、記録の分け方、ステーション制という骨格は4つの型に横転可能
- 「効果」を人間に固定し、エージェント自己採点の過信を防ぐ構造は普遍的

**失敗している点:**
- ビルド・テスト・QA環境の定義がゲーム開発に強く依存(型B/Cでは不要 or 意味変化、型D では既存テストスイート前提)
- ステーション定義(特にコンテンツ番)がゲーム開発に最適化
- フェーズの概念が新規開発を前提(型D の保守作業では不自然)
- 「テスト数の単調増加が正常」という指標が型ごとに妥当性が異なる

**修正案の実装性:**
修正案1〜8は分岐を避け、**型選択フロー + テンプレート多重化 + セクション化**で吸収可能。
既存テンプレートを壊さず、型別ガイドを付属させる形式が最小改変。
