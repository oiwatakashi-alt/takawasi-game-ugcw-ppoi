# ゴール駆動ループ 汎用テンプレート v2.0

作りたい物ごとにこのフォルダを丸ごとコピーし、[ ] を埋めて使う。
Codex goalモードと Claude Code /goal・/loop・/batch の両方で駆動できる。
v2.0はsaikyo-workflow実走行(2026-07、43ターン)の失敗観測に基づく構造再設計
(何をなぜ変えたかはCHANGELOG.md v2.0)。

## 設計原理(5つ。v2.0で改訂)

**1. 注ぐ、指さない。** 規則はファイルに置いて「読め」と命じても、読む義務自体が
脱落する(43ターン中23ターンで宣言が無言脱落した実測)。毎ターンの規則は
session_brief.shが現在番の分だけ逐語で注入する。常駐は薄く、注入は狭く。

**2. 順序制御はゲート、判断はMD。** 同じ契約でも、機械検査される成分は34/34で守られ、
散文の成分は0/23だった。配車はcheck.shが計算し、宣言はその引用。turn_close.shが
ターン契約を検査する。ただし計器(検査)と行動機械(常駐・自動配車)は別物——
後者は既定で作らない(00_MISSIONの機構関門)。

**3. 指揮と作成の分離。** 親は判断・配車・検収だけを持ち、作成は既定でワーカーに
委譲する。自分で作った文脈では自分の成果を客観判定できない(作成バイアス)。
ワーカーは80%でよい——完全性でなく、独立レビューと再試行の安さに投資する。
委譲の発火は計画の実行者フィールドで配線する(美徳に頼らない)。

**4. 採点者の分離。** 自己判定は「存在/動作」まで。「効果」は人間、UIUXは市場アンカー。
ワーカー報告は定型契約+機械検札+抜き取り監査。レビューの脱相関は人格でなく
情報状態を変えることで作る。

**5. 成果で閉じる。** 完了条件は機構の緑(動いた)でなく成果の緑(使われた)まで書く。
機構は消費者と退役条件を持って生まれる。監査番は毎回、削除候補を最低1つ指名する
——加算は毎ターン起きるが、減算は儀式にしないと永遠に起きない。

## 運用サイクル

1. 00と02を埋める(最初は薄くてよい)。`git init` + 初回commit
2. `START.md` をエージェントに指してループ開始
3. 完了条件を満たすと停止 → 検収(検収パックの証跡で判定)
4. 照合ログ記録 → 計画改訂 → 次フェーズのゴール文(北極星を逐語引用)→ 再起動
5. 文脈が濁ったらセッション再起動プロトコル(03_DISPATCH末尾)

## 記録層の分担(混ぜない)

- **git** = 事実(ターンごとcommit、書式 `[番名] 要旨`)
- **04_STATE** = 現在(常に現在形)/ **08_LOG** = 経過(追記専用)/
  **07_KNOWLEDGE** = 調べたこと / **09_LEARNINGS** = やり方の学び /
  **plans/** = ウェーブ計画の履歴 / **docs・決定ログ** = 決定と経緯

## 最小構成で始めたいとき

小さい物なら00(北極星と柵)と02(ゴール文)だけで開始し、フェーズ2つ目で残りを足す。
テンプレートに従うこと自体が目的化したら削る——この文書群は成果物のための間接費であり、
それ以上ではない。儀式的に完璧なターンは進捗に見えるが、進捗は成果物の側にしかない。
