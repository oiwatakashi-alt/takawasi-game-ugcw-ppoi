# 03_DISPATCH.md — 配車と番の定義(順序制御はゲートに、規則は注入で)

v2.0の設計原理: **配車は宣言でなく計算**。散文の手順規則は作業の勢いに負けて
静かに脱落する(同一ステップ内でゲート付き成分の遵守34/34、散文成分0/23——CHANGELOG v2.0)。
だから順序制御は計器(tools/check.sh)が計算し、番の規則はブリーフ(tools/session_brief.sh)が
逐語で注入する。エージェントの仕事は「計算結果を引用して、注がれた規則だけで働く」こと。

## 毎ターンの最小核(これだけがターンの義務。全て機械検査される)

1. `bash tools/session_brief.sh` を実行(STATE・計器・**配車結果**・現在番の規則が注入される)
2. 配車結果を引用して番を宣言(書式: `番: [名前] / 根拠: [check.sh出力の規則番号]`)。
   引用のない配車は無効
3. その番の規則(注入された7本以内)だけで働く
4. commit(書式 `[番名] 要旨`)+ 04_STATE.mdを欄置換 + 08_LOG.mdに完了報告
5. `bash tools/turn_close.sh`(番タグ・STATE・LOG・報告行数を機械検査。赤はamend)

予算超過(規則が7本を超えて同時に要求された)と感じたら、落とす規則を選んで
**宣言して捨てる**(LOGに1行: 「予算超過につき○○を落とし次工程へ」)。黙って落とさない。

## 配車規則(check.shが同じロジックで計算する。この表は文書側のミラー)

上から評価し、最初に該当した番に入る:

1. セッションの最初のターン → **偵察番**
2. check.shの計器に**赤が1つでもある** → **監査番**(赤の種類を問わず。
   推奨番と赤判定の不一致は事故の元——赤は常に監査が拾う)
3. フェーズ完了条件の充足を申告する/フェーズ予算(消費ターン>目安)を超過した → **監査番**
4. 完了ウェーブ数が再計画トリガーに達した → **監査番**(再計画議題)
5. どれにも該当しない → **実行番**(現ウェーブ/現マイルストーンの指定に従う)

番は3つしかない。作業の種類(実装/リサーチ/整備/品質/コンテンツ)は番でなく
**マイルストーンの「型」フィールド**(01_PLAN)が持つ——種類分けの規則は計画が運び、
ターンは実行するだけ(型ごとの心得は下の実行番に含む)。

## 偵察番(セッション初回のみ。規則5本)

1. ブリーフのSTATE鮮度が黄・赤ならSTATEをgit側の事実に合わせて再建する(gitが正)
2. 北極星(00_MISSION)を一度逐語で復唱する(セッション単位のアンカー再設置)
3. 未処理キュー(hub/mail・inbox・user/drop・humanops)の対応要否を判断する
4. 現状の一行要約と次番を宣言する
5. このターンでの実装は禁止

## 実行番(既定。規則7本)

1. 現ウェーブ/現マイルストーン(01_PLANの行)のスコープ内だけで働く
2. **実行者フィールドに従う**: ワーカー委譲が指定されていれば、ブリーフ(05_CONTRACTS)を
   書いて委譲し、指揮は検収に回る。親直営に変えるなら逸脱として理由をLOGに宣言する
   (委譲は美徳でなく計画の指定で発火させる——発火実績5/5の配線。CHANGELOG v2.0)
3. 受入基準(検証コマンド・テスト・証跡)が緑になるまでがスコープ。テスト・実機QAは
   実行の一部であり、後回しの別ターンにしない
4. 検証が失敗したまま先に進まない(stop-and-fix)
5. 範囲外の改善は実行せずSTATEの発見リストへ(例外: データ損失・安全性は即時フラグ)
6. リサーチ型のマイルストーンでは、調べる前に07_KNOWLEDGE.mdを検索し(再走防止)、
   結果は状態タグ付きで着地させる(着地先のないメモで終わらせない)。
   市場レビューを数えるときは06_REVIEW_FILTER.mdを必ず通す
7. 委譲した報告を判断に使ったら、LOGの「報告消費」欄にどの報告かを書く
   (消費の記録がない機構は監査番の退役候補になる)

## 監査番(境界と異常時。規則7本)

1. check.sh出力を材料に、赤の原因を最初に処理する
2. 自分の直前ターンの作業は判定しない(最低1ターン空ける)
3. 完了判定は証跡ベース: 01_PLAN・02_GOALの完了条件をgit・証跡と突合する。
   **機構を作ったフェーズでは消費の観測(出力が判断に使われた実績)を判定に含める**。
   停止・完了を申告するときは `bash tools/stop_check.sh` の棚卸しを引用し、
   残存項目ごとに正当化(blocked/休眠/過剰側)を書く——残存の主張は機械の棚卸しで置換する
4. **削除候補を最低1つ指名する**(「削除候補なし」も可だが明記する。加算は毎ターン
   起きるが減算は儀式にしないと永遠に起きない)。機構の退役条件との照合もここで行う
5. 北極星照合(このフェーズは北極星に向かっていたか一行)+常駐入口ファイル
   (CLAUDE.md/AGENTS.md/00/02)が現フェーズの実態と食い違っていないかの同期突合
6. 抜き取り検査: 直近のワーカー報告から1本を選び、原本(成果物・証跡)と突合する。
   誤りが出たら同ワーカーの直近成果を全数再検収し、ブリーフ雛形側の欠陥を疑う
   (雛形は誤り増幅器——1行の欠陥×N実行になる)
7. 09_LEARNINGSの未振り分け行を刈り取る/判定を照合ログ(01)に記録する

### 再計画議題(規則4で入ったときだけ、上記に代えてこちら)

⓪00_MISSIONと突合(第0層・撤退条件・ラチェット照合。触る案は人間へ)
①`bash tools/wave_stats.sh` で見積もりvs実績を較正 ②STATEの仮定欄の棚卸し
③09_LEARNINGSの刈り取り ④次のウェーブ群を価値順に並べ01_PLANを改訂
⑤ルーチンの割り出し(判断ゼロの反復手順→計器化候補として09へ)
⑥減算レビュー: この期間に足した機能・機構を列挙し、北極星(柱)への寄与を一文で申告。
言えないものは削除候補へ降格提案。[網羅]タグは対象外。あわせて対の加算儀式として
軸生成工程(下の節)を1回実行する。再計画自体も1ウェーブとして採番する

## 軸生成工程(燃料の自給——収束探索だけでは湧かない生成軸を作る)

棚卸しと既存作比較は既存の人工物に錨を打つため**収束的**で、これだけだと燃料が
尽きて見えるたびにオーナーの起点投入待ちで止まる(収束探索病)。第3の燃料源:

1. 発火条件: 候補置き場が枯れた/stop_checkが緑を出した直後(停止判定の前に必ず1回)/
   再計画議題⑥
2. 入力: **飢えてる軸**(完了実績を軸・領域別に数えて最少のもの)×**到達可能な手段クラス**
   (07_KNOWLEDGEの能力台帳)×**北極星**
3. 生成: 飢え軸×手段ごとに**生成器**(個別ネタでなくパターン族。族名/機能型/量産単位を
   1行)を3〜5案。この段階で実現可能性・工数で捨てない(生成と評価の分離)。
   仮採点は北極星への寄与一文のみ
4. 選別: 上位数案をオーナーに**メニュー形式で提示し、拒否権の行使だけ求める**
   (発明でなく選別に人間の注意を使う)。運転嗜好が継続なら、無応答時は市場アンカー
   照合で1案を自走採用し、採用理由を台帳に記録する
5. 採用された生成器は候補置き場に**族として**登録し、要素ごとの個別課題を湧かせる
6. 生成も空振りしたら、その証跡をLOGに残す——**「全候補が尽きた」はこの証跡があって
   初めて言える**

## ウェーブ運用(複数ターンの塊だけ)

**1ターンで終わる作業はウェーブにしない**(開閉の儀式がターン自体より高くつく——
1ターンウェーブ13本の観測。CHANGELOG v2.0)。目安2ターン以上の
マイルストーン貫通だけ `bash tools/wave_open.sh 名前` で開き、受入緑+実績記入で
`bash tools/wave_close.sh`。計画MD(plans/)は上書き・削除しない(見積もり較正の原資)。

## セッション再起動プロトコル(文脈が濁ったら)

同じことを二度探した/台帳にない事実を記憶から書きそうになった/指示と古い決定が
食い違った——いずれかが起きたら: 04_STATE.mdを現在形に書き直し→commit→
新セッションで偵察番から。持ち出すのは結論と根拠であって経緯ではない。

## 並列レーン(任意・上級)

独立した縦切りが複数あるときだけ。所有権(ファイル・ディレクトリ)を重複させない。
合流は監査番がSTATE上で行い、レーン間で生出力を直接渡さない。
