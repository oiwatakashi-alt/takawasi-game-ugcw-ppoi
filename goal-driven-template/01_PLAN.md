# 計画 — Takawasi Game

計画のsource of truth。フェーズごとに改訂する。エージェントが計画から外れる判断は
自由だが、外れたことは監査番が照合ログに記録する。
**委譲はこのファイルの実行者フィールドで発火する**——マイルストーンに実行者が
書いてあるとき委譲は起き、書いていないとき起きない(観測済み: 発火5/5はすべて
計画に実行者指定があった場合。CHANGELOG v2.0)。

## ルーブリック(品質・面白さの外部アンカー)

「良いか」を毎ターン再解釈させないための固定表。基準線のリサーチから起こし、
リサーチ型マイルストーンの出力で育てる。

| 参照元の価値要素 | 対応する自作システム | 存在/動作の確認手順 | 実機での見た目・挙動(証跡) | 効果の判定者 |
|---|---|---|---|---|
| 戦略結果が次の戦場を変える | 五帯戦線、主戦場1件、小作戦3〜4件、pressure/resource carryover | `FILEMAP.md` route/data map照合 + `npm run build` | 既存 `outputs/` に戦略→戦闘→戦果→次ターンのQA報告あり。現フェーズで新しい一周操作ログを採る | 人間(チェックポイント) |
| 常時microでなく指揮判断をする | StandingOrders、戦線、予備、施設、pause、staff recommendation | battle/campaign型と保存経路の照合 + 対象QA | 既存QAは個別機能中心。全体の動的手触りは再審査未了 | 人間(チェックポイント) |
| 損耗と成長が同じ軍団へ残る | unit/officer XP・疲労・負傷・教訓・equipment/fortification history | save v8の保存/読込とAfter Action carryoverを実操作 | 既存 `outputs/*carryover*` 報告あり。効果判定「もう一度続けたい」は未記録 | 人間(チェックポイント) |
| マップ優先で読みやすい戦況 | UGCW参照の戦略/戦術UI、固定HUD、戦線/目標/脅威layer | 基準線メモとdesktop実画面を並べて照合 | `WEBリサーチ/06_*` `07_*` と既存スクショ。総和密度の再評価は未了 | 市場アンカー照合+人間 |

原則: エージェントの自己判定は「存在/動作」列と「実機での見た目・挙動」列まで。
実機列は実際に起動・操作した証跡(スクショ+再現手順)がある場合のみ埋めてよい。
「効果」列(面白いか・価値があるか)はフェーズ境界で人間が判定する。
例外: UIUXの行は判定者列に「市場アンカー照合」と書き、基準線画像+レビュー根拠との
照合で判定する(エージェント実行可)。製作者の趣味はUIUXの採点者にしない。
証跡は撮る側が選べるため、各証跡に再現手順を必ず添える。
体験が時間軸を持つ場合(動き・音・操作感)、**静止画は動的欠陥を検出できない**——
フリーズした画面の1フレーム目が「正常」としてQAを通過した実測がある(フリーズ・
モーションの貧弱・反復感・総和のうるささは静止画に写らない)。実機列の証跡は
録画または実操作ログを既定とし、スクショは構図・色・レイアウトの判定にだけ使う
(根拠: CHANGELOG v2.7)。

## コア方式ゲート(制作系プロジェクト——コアループ・表現方式の選定と再審査)

ジャンル・コアループ・表現方式・課金方式は、00_MISSION第0層に明記されない限り
検証仮説である(00_MISSION第2層の例外既定)。根拠: CHANGELOG v2.7(Callcrew停止判断)。

- **優先順位の既定**(00_MISSION「迷った時の優先順位」の制作系プロジェクト既定):
  1. 制作主体(現行LLM+利用可能ツール)で、競合と並べて破綻が目立たない水準まで
     整合して仕上がること
  2. 儲かること(市場層・訴求。ただし商品の質に対する対価として)
  3. 使う人が楽しい・価値を感じること(質そのものを上げる)
  2・3が基準未満のとき、追加コンテンツで埋めるのは禁止——コア方式・ジャンルごと
  再選定する(破壊的変更を強く見る)。**既存実装量は続行理由として無効**。
- **選定は複数案比較**: 制作適合性のプロファイルが異なる案を最低3つ並べてから選ぶ
  (例: リアルタイム動作型/低モーション選択型/一枚絵・物語型)。比較表に
  **受入の機械検証可能率**の列を含める——機械ゲートを作れる領域は安く確実に仕上がり、
  作れない領域(視覚品質・動きの魅力)は欠陥が人間の目まで生き残る(実測)。
  発案したインスタンスに採否判定をさせない(三窓レビュー盤か新品窓で反証)。
- **最小実証が量産の前提ゲート**: コアループ1周が実際に操作でき、効果判定者
  (原則人間)が一度「繰り返したい」を通過するまで、コンテンツ量産フェーズ
  (ステージ・面・記事等の複製的増産)に入らない。証跡は企画書・静止画でなく
  **実操作+短い録画**。効果判定がblockedのまま量産した分は、コアが覆ると全損する(観測)。

## 柱×視点マトリクス(多柱並立型のみ。単柱プロジェクトはこの節を削除してよい)

柱の宣言と委任弁の定義は00_MISSION.md。検収の問いは柱ごとに書く。

| 柱 | 面白さの問い(この柱が好きな人にとって) | 苦手な人の弁(委任・自動化) | 柱間の噛み合い |
|---|---|---|---|
| 動的戦略 | 今の位置と圧力から、主戦場・小作戦・撤退の選択が毎ターン変わるか | 小作戦auto-resolve | 選択した戦場・偵察・資源が配置と敵波へ流れ、戦果が次ターンpressureへ戻る |
| 半自律戦術 | 戦線と命令方針を決めた結果が、直接microより大きな意味を持つか | pause、推奨、保存方針、一括指示 | 戦術の役割・失敗がAfter Actionと次戦教訓へ流れる |
| 軍団の履歴 | 同じ部隊・将校を守るか消耗させるかで迷い、履歴に愛着が出るか | 推奨配属・教訓方針・plan set | 戦果が能力・疲労・装備・政治costになり、次の戦略/戦術選択を変える |
| 防御長期戦 | 陣地へ投資するか捨てるかが、その場の最適化で終わらないか | 施設効果比較と修理候補表示 | fortification・補給・医療が戦闘損害と次ターン資源へ循環する |

検査規則: 噛み合い列が空の柱=詰め合わせキメラの兆候。弁列が空の柱=全ユーザーへの強制。
どちらも監査番の議題。

## UI/UX方針(UIを持つプロジェクトのみ。無ければこの節ごと削除)

### UI媒体の決定(実装前に一度。決定ログと同形式で記録)

Webを既定にしない。候補: Web / ネイティブGUI / TUI / 既存ツール上 / 静的画像・PDF。
選択肢を並べる前に基準を書く(配布先・オフライン要否・描画性能・入力の種類・
乗り換えコスト——乗り換えコストは系統的に過小評価されるため必ず含める)。

- 決定: local-first desktop Web(Vite/React) / 理由: 現在の全ドメイン・保存・QA資産がこの媒体で一周貫通している / 敗北条件: 大規模戦闘の動的表現・入力・配布要件がWebでは競合水準へ届かないとコア方式ゲートで実証されたとき

### UI表面の二層(適用する規則が違う)

- **義務層**: 業務・目的の達成に必要な全項目・全状態の露出([網羅]スコープのUI面)。
  判定者は網羅表との突合。密度が上がり不格好になっても網羅が優先
- **魅力層**: それ以外の表面。基準線画像アンカー・UX細部台帳・濃度管理の適用域

### 基準線画像アンカー(UIUXの外部基準)

- 名作の選定基準を先に書く: [例: 同ジャンルで販売数上位、評価85%以上、複数タイトル]
- 選んだ名作のスクショを基準線画像置場(00_MISSIONの定数)に保存し、実装・QAのたびに
  視覚アンカーにする
- **評価箇所の注記が必須**: 各画像に「どの要素がなぜ評価されているか」を大衆レビューの
  逐語引用+言及数で注記する(言及数は06_REVIEW_FILTER.mdのフィルタ通過後の数)
- 参照は分析用。アセットの模写・流用はしない

### UX細部台帳(好評体験の採取→反映→確認)

| 出典(URL/日付/言及数) | 評価されている細部(逐語) | うちでの対応 | 状態 |
|---|---|---|---|
| `WEBリサーチ/06_battle_screen_visual_reference.md` / 2026-06-30 | map-first、固定HUD、盤面の読みやすさ | 戦術マップ+固定情報帯へ反映 | 実装済・v2.7動的再審査中 |
| `WEBリサーチ/07_campaign_and_army_screen_visual_reference.md` / 2026-06-30 | 戦略/Army/Armory/Barracks/Careerの画面分離 | Theater/Camp tabsへ反映 | 実装済・v2.7総和再審査中 |

- 候補→採用は機械判定([N]件以上の独立言及 or 名作[M]作共通)。実装済→検収済は
  基準線画像・レビュー根拠との照合一致で昇格。棄却も行を消さず状態だけ変える
- **反転採取(負の行)**: 不満は好評と同格以上の証拠。診断力順=名作への不満
  (推奨レビュー内の不満が最上位)>低評価作のUIUX特定不満。負の行は
  「対応=やらない+検収=アンチパターン不在の確認」で台帳に入れる
- **市場アンカーの例外(恒久)**: 搾取的収益化は市場で勝っていても採らない。
  市場アンカーは品質の判定者であって倫理の判定者ではない
