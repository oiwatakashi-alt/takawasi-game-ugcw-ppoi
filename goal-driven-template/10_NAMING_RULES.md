# 10_NAMING_RULES.md — 命名規則(ベースルール)

目的: grep・find・lsの出力**だけ**でLLM(と人間)が構造を理解できる状態を保つ。
名前は「開かなくても読める唯一のメタデータ」——ファイルを開く回数を減らすことが
文脈経済の最上流の対策。多少の冗長は許容する(役割が語れない短さより冗長が勝つ)。

## 原則(すべての新規ファイル・フォルダに適用)

1. **役割を名前で語る**。中身を開かないと分からない名前(misc, tmp, data2, 新しいフォルダ)は禁止
2. 冗長は許可、ただし上限: 役割_対象_修飾の**3〜4要素まで**。文章型の名前にはしない
3. 同じ概念には同じ語(検証をverify/check/testと揺らさない。迷ったら既存ファイルの語彙に合わせる)
4. 読み順・優先順に意味があるなら番号プレフィクス。ループ核は2桁(`NN_`)、ウェーブ計画は3桁(`NNN_`)
5. 時系列アイテム(起票・投函・証跡・スナップショット)は日付プレフィクス
   `YYYY-MM-DD(_HHMM)_`——名前ソート=時系列になる
6. 方向を持つものは `to-〔宛先〕` / `from-〔差出〕`
7. 言語: ループ核・ツール・フォルダはASCII(スネークケース)。人間が読む起票・計画・報告の
   件名部分は日本語可。その層の既存慣行に合わせる
8. スペース禁止。拡張子は中身と一致させる
9. **中身が変わって名前が嘘になったら改名する**。ただし改名は必ず参照の全更新とセット
   (下の改名手続き)

## 層別の規約(散在していた慣行の集約。各READMEと矛盾したらこちらが正)

| 層 | 規約 | 例 |
|---|---|---|
| ループ核 | `NN_役割.md`(2桁=読み順) | `03_DISPATCH.md` |
| 特殊ファイル | 大文字1〜2語(入口・生成物・履歴) | `START.md` `DASHBOARD.html` `CHANGELOG.md` |
| ウェーブ計画 | `plans/NNN_名前.md`(3桁連番) | `plans/012_補充システム.md` |
| hubへの投函 | `YYYY-MM-DD_HHMM_差出_件名.md` | `2026-07-06_0900_cowork_検証依頼.md` |
| レポート | `〔入力ファイル名〕_report.md` | |
| humanops起票 | `YYYY-MM-DD_件名.md` | `2026-07-06_VPS契約.md` |
| 反証 | `〔元ファイル名〕_反証.md` | `市場規模メモ_反証.md` |

## 機械検査可能なサブセット(違反が観測されたらcheck.sh系に降ろす候補。今は作り込まない)

スペース含有 / hub投函・humanops起票の日付プレフィクス欠落 / inbox直下(to-*外)への投函

## ループへの組み込み

- **作成時**: どの番でも、新規ファイル・フォルダはこの規則で命名する(00の不変の規律に登録済み)
- **点検**: 整備番が命名ドリフト(規則違反・語彙の揺れ)を監査し、違反は改名候補として
  STATEの発見リストへ(その場で改名しない——改名は下の手続きで)
- **改名手続き**(小さくても大規模改変として扱う):
  1. `bash tools/filemap_snapshot.sh 〔ラベル〕` でファイルマップ履歴を取る
  2. 改名は1ウェーブとして実施(plans/に計画。委譲可——ブリーフの参照にこのファイルを渡す)
  3. 参照の全更新とセット: 旧名でgrepし、残存0(CHANGELOG・filemap-history内の履歴記述は除外)を
     確認してから閉じる
  4. commit書式: `[整備] 改名: 旧→新(参照N件更新)`

## 外部環境の作法(観測済みの失敗からの規則。saikyo-workflow走行 2026-07で全件実測)

- **bash 3.2(macOS標準)**: 変数展開の直後に区切りなしで日本語等の非ASCIIが続くと
  誤動作する既知の癖がある。tools/配下のスクリプトでは常に `${VAR}` 形式で書く
- **常駐登録の解除セット**: 常駐プロセス(launchd/cron等)を起動するスクリプトを削除する
  ときは、OSレベルの登録(plist・crontab行)も同じ変更で解除する。孤児化した登録は
  残骸でなく事故(平文キーの残留等)に発展する
- **launchdの監視方式**: `.gitkeep`運用のあるフォルダを監視する場合、
  `QueueDirectories`(非空で発火)は即暴発する。`WatchPaths`(変化イベント)を使う
- **常駐watcherがいる環境のテスト**: 「テスト投函して即rm」は成立しない(投函から
  数秒で実処理が走る)。テストは本番パイプラインごと流し、生成物を受入証跡に転用する
- **秘匿情報**: plist・スクリプト・ログに平文で置かない。secretsファイル
  (chmod 600・gitignore対象)に分離し、台帳・ブリーフからは所在参照で指す

## 大規模改変のためらい除去(この規則の目的の半分)

命名・構造の大改変を躊躇しない。中身の履歴はgitが持ち、意味の地図はfilemap-historyが持つ
——**両方あれば大改変は可逆であり、可逆な決定は速く大胆に下してよい**。
腐った構造への継ぎ足しの方がリネームより高くつく。
義務は一つだけ: 改変前に `tools/filemap_snapshot.sh` を実行すること。
