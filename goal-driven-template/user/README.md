# user — ユーザー提供資料の置き場と検証規約

ユーザーが投入する条件・資料・参考文書の受け皿。**優先度は高い、しかし絶対視しない**——
ユーザー提供の資料も間違っている可能性を常に疑う(検証対象)。
ただし削除は絶対にしない(ユーザーの所有物。移動と記録のみ)。

## フォルダの意味(状態機械)

| フォルダ | 状態 |
|---|---|
| `drop/` | 投入(未処理)。ユーザーはここに放り込むだけでよい。ループが自動で拾う |
| `sources/` | 昇格済み。検証を通り、計画・実装の根拠に使ってよい資料 |
| `rejected/` | 再検討(却下)。疑義があり差し戻したもの+反証記録。**最終処分ではない** |

## トリアージ(偵察番が検知し、型で振り分ける)

drop/ の各ファイルを3つの型に分ける:

1. **指示・希望・好み** → 検証不要(ユーザーの領分。解釈・目的・好みは検証対象にしない)。
   01_PLAN/02_GOALの計画に写像してから sources/ へ
2. **外部事実の主張**(数値・技術情報・市場データ) → リサーチ番で検証してから昇格。
   検証結果は07_KNOWLEDGEに状態タグ付きで登録し、資料は sources/ へ
3. **参考資料**(記事・仕様書・過去の調査等) → 出所と鮮度を確認して昇格。
   荷重する引用は原文と照合する

## 優先順位の規則

- ユーザー資料は外部Web情報より優先度高め(依頼の文脈を最もよく知る証拠クラス)
- ただし**独立に検証できる事実**が外部証拠と衝突したら、証拠の強い側を採る。
  ユーザー資料が負けたら rejected/ へ——気を使って黙って従うのは禁止。
  間違った前提での作業は、量と質に関係なく無効になる
- 判断に迷ったら: 解釈と好みはユーザーが常に正しい/検証可能な事実は証拠が正しい

## 反証・反論の記録(rejected/ の規約)

却下時は元ファイルを rejected/ に移し、隣に `〔元ファイル名〕_反証.md` を置く:

```
# 反証: [元ファイル名]
- 元の主張(逐語): [パラフレーズ禁止]
- 矛盾する証拠: [内容+URL/検証方法。ユーザーが追跡できる形で]
- 確度: [レンジ併記 例: 高い(70〜90%目安)+根拠一行]
- ユーザーへの問い: [何があれば再検討できるか]
- 日付/番: [ ]
```

反証はダッシュボード(DASHBOARD.html)に表示され、ユーザーはそこで確認して
再提出・裏付け追加・異議で応答できる(異議は drop/ に入れる——ユーザーの異議も検証対象。
往復の記録は反証ファイルに追記していく)。
