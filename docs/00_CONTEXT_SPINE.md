## Implementation Recovery Rule

Before implementing a non-trivial feature:

1. Check `FILEMAP.md` for the expected address.
2. Check the closest requirement/spec doc for intent.
3. If the chosen address changes, update `FILEMAP.md`.
4. If the feature changes product behavior, update the relevant docs, not only code.
5. If an image need appears, add it to `docs/09_IMAGE_GENERATION_BACKLOG.md` before generating.

Issued commands are preserved as battle-only `commandTransmissionEvents`, then `BattleResult.commandTransmissionOutcomes` shows `伝令評価` in After Action and writes the same delay/congestion/arrival records into campaign last message, unit history, brigade-commander history/fatigue/XP, chief-of-staff warning history, existing staff recommendation `前戦評価`, and low-weight next-battle tactical lessons. Command-post fatigue now closes the loop from After Action staff outcomes back into the next battle. `src/game/battle/commandPost.ts` derives a shared battle-only `commandPost` profile from the assigned `参謀長` status/fatigue; `orders.ts` uses it to add transmission-delay reasons such as `司令部疲労+2秒` and to reduce queued-command handling capacity for `一括混線`; Battle Command shows the command-post label in the top HUD and its reason inside `予約指揮`. Deployment also shows the same profile before battle as `司令部伝達` and raises `司令部伝達警告` with a `将校調整へ` route when the next battle will suffer transmission delay/capacity loss. Desktop QA is recorded in `outputs/takawasi-command-post-fatigue-qa-report.json` and `outputs/takawasi-deployment-command-post-warning-qa-report.json`; mobile QA remains outside the active target.

Deployment command issue planning is now part of the battle-preparation package. `DeploymentBattlePlan` and `StandingOrderPlanSet` can store `標準発令`, `分割発令`, or `逐次発令`; Battle creation carries the choice into `BattleState.commandIssuePlan`; Battle Command displays it in `予約指揮`; and `commandCongestionReport` uses it to cap effective batch size for split batches or penalize multi-command batches under strict direct handling. Deployment also computes a `伝令運用` advisor from expected batch size, command/organization doctrine capacity, and command-post fatigue, then recommends standard/split/direct handling with reasons and a one-click adoption button. Battle Command mirrors that decision during combat with a compliance line (`方針待機`, `方針適合`, `分割適用`, `混線注意`, `逐次違反`) and now lets the player execute the policy directly: split/direct buttons issue only the first allowed batch and keep the remaining queue for later, with congestion calculated from the actually issued batch.
