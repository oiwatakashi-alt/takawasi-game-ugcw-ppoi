# UGCW vs Takawasi Game — 戦闘画面・戦術システム・軍アニメーション比較

調査日: 2026-07-10  
状態: 実装停止。VPS変更なし。次の整形方針を決めるための比較資料。

## 結論

現状の主問題は「背景画像が足りない」だけではない。

`takawasi game` は、地形、戦線、施設、目標、敵指揮、命令遅延、撤退予測、戦果持越しまで、戦術システムの種類はかなり揃っている。一方で、1280x720の画面では、UGCWが戦術マップを主役にしているのに対し、現状はマップ上の部隊カードと、マップ下に続く長い詳細UIが主役に見える。さらに、部隊の位置更新・行動状態はシステム側にあるが、画面上のアニメーションは「移動中の破線枠」や状態文字が中心で、軍が動いて撃ち、崩れ、退くという因果が視覚的に伝わりにくい。

したがって次に必要なのは、いきなり大きな背景絵を生成することではない。

1. 戦術マップを画面の主役に戻す。
2. 右/下の詳細を要約→選択詳細へ折りたたむ。
3. 部隊の状態を、位置・隊列・射撃・被弾・後退の短い動きとして見せる。
4. その後に、地形・部隊・煙・着弾の小さな画像アセットを足す。

## 1. 戦闘画面の比較

| 観点 | UGCW | Takawasi現状 | 判定 |
|---|---|---|---|
| 画面の主役 | 手描き風の戦術マップが大部分。地形・道路・川・森・陣地と隊列を先に読む | マップは存在するが、上部の警報/予測、マップ上の大きな部隊カード、下部の旅団詳細が縦に連続する | UI再編が先 |
| 上部情報 | 日時、戦力比、任務時間、目標条件など、大局を短く固定 | 参加旅団、視界、発見、隘路、司令部、地形などが横並び。情報は多いが優先順位が弱い | 要約化 |
| 選択部隊 | 左下のUnit Cardに選択部隊の状態を集約 | 下部に旅団カードを複数並べ、選択・命令・自律理由・射撃監査を同時に出す | 選択1件を主表示、他は短縮 |
| 命令 | 下部の固定Command buttons。部隊選択に応じて変わる | 保持、前進、側面、休息/補給、築城/修理、後退に加え、戦線、施設、敵指名、火力計画、予測が縦に広がる | コア命令と詳細命令を分離 |
| 目標 | Objective Conditionsを押して勝利/引分条件を確認。目標状態は常に追跡できる | 勝利/補給/視界点、イベント、敵波、戦線維持を実装済み | システムは強い。見せ方を圧縮 |
| ミニマップ | 戦場の全体把握とカメラ移動を支える | ミニマップとジャンプ操作は実装済み | 維持。ただし本マップとの役割を明確化 |
| 速度/停止 | 速度操作と一時停止が常設 | 停止、1〜3倍、撤退を常設 | 維持 |
| 保存 | 公式ガイドは戦闘中のSave/Load/Restartを明記 | 既存probeではTheaterの保存は保持したが、Battle reloadはTheaterへ戻った | B契約の最優先未達 |

UGCW公式ガイドはBattle interfaceを、Menu、Date/Time、Force Bar、Mission Time、Objective Conditions、Unit Card、Army icons、Command buttons、Camera、Mini Map、Speed buttons、Action messagesに分解している。これは機能数の比較表ではなく、画面の固定位置と読む順番の設計として重要である。現状は同じ種類の情報を多く持つが、詳細パネルが主戦場の視線を奪っている。

## 2. 戦術システムの比較

### UGCW側で確認できる強み

- 部隊を個別に直接操作するか、師団AIへ任務を渡すかを選べる。
- 防御線や移動経路を描き、隊列単位で任務を実行させる。
- 部隊カードの兵数、損失、再装填、補給、士気、遮蔽、状態を、地形上の部隊と結び付けて読む。
- 道路、川、橋、森、高地、陣地が、単なる背景ではなく視界・遮蔽・移動・防御の判断へ接続する。
- 戦闘結果で兵士、将校、評判、次の戦場の条件が変わる。

公式ページも、個別部隊への命令、師団AI、防御線/側面移動の描画、地形利用、補給防衛、敵AIの弱点攻撃と退却を主要要素として挙げている。公式ガイドでは、Moraleが崩れると部隊が制御不能になりRetreatへ移行すること、Coverが地形に依存すること、Conditionが効率・近接・速度に影響することまで記載されている。

### Takawasi側の実装済みシステム

現状のコードには、UGCWの抽出を超えて次の層がある。

- 戦線区画、基準位置、後退線、戦線スケッチ。
- 旅団ごとの保持/前進/側面機動/休息補給/築城修理/後退。
- 地形の遮蔽、射線遮断、射程減衰、泥濘、稜線。
- 勝利地点、補給点、視界点、施設、防衛耐久。
- 敵群の突撃軸、突破、側面、凝集、指揮階層、指揮源喪失、士気状態。
- 命令予約、伝令遅延、指揮容量、火力計画、標準自律方針。
- After Actionでの損耗、経験、疲労、装備摩耗、戦術教訓の持越し。

したがって、現段階で「戦術システムを増やす」ことは優先ではない。システムが既に多いため、画面上で一度に全部を読ませるほど、UGCWのような戦術盤面の軽さから離れていく。

## 3. 軍のアニメーション/アクション比較

### UGCWから確定できること

静止画は、隊列、旗、地形、煙、射撃の結果らしき表現、部隊密度を観察するのに使える。しかし静止画だけでは、移動の滑らかさ、射撃周期、隊列の向き直り、被弾時の反応、退却の見え方は判定できない。

公式資料からは、次の「動きの意味」は確定できる。

- プレイヤーは移動経路を地形上へ描く。
- 部隊は防御線・攻撃目標・側面経路を任務として受ける。
- 部隊は地形、補給、士気、Conditionの影響を受ける。
- 師団AIは大きな軍を任務単位で動かす。
- 弱点を突かれた部隊は退却し、指揮不能状態も起きる。

つまり、UGCWのアニメーション価値は、派手なエフェクト単体ではなく「命令→隊列移動→接敵→射撃/被弾→士気/補給変化→再命令」という因果が短い時間で見えることにある。

### Takawasi側の現状

`resolveTick.ts`には部隊位置更新、移動、射撃、交戦、敵波、敵指揮、士気/状態更新のロジックがある。`BattleCommandScreen.tsx`も、部隊の`position`、`isMoving`、`actionReason`、`fireMissionId`、敵の`phase`、`commandState`、`moraleState`を表示している。

ただし画面表現には以下の差がある。

- `.map-unit`の最小幅が大きく、部隊が「隊列の駒」ではなく「長い情報カード」に見える。
- `isMoving`時の主な視覚差は破線枠で、位置補間・進行方向・隊列の向き直りが明確ではない。
- `volleying`、`targeting`、`phase-breakthrough`などの状態クラスはあるが、発砲、着弾、煙、衝撃、退却を連続した短いモーションとして読む設計が弱い。
- マップ上の部隊名/人数/状態文字が多く、動きそのものより説明文を読む時間が長い。
- 画面録画で初めて分かる動的品質を、既存のスクリーンショットだけでは評価しきれない。

ここは「アニメーションがない」と断定する段階ではない。ロジック上の状態遷移はあるが、UGCWのように視覚的な軍の動作へ変換されているかを、短い録画と状態別の観察表で検証する段階である。

## 4. 優先順位

### P0 — 戦闘画面の面積と読む順番

1280x720の検収基準を次で固定する。

- 戦術マップを最初の視線の中心に置く。
- 上部は「戦闘時間/戦線維持/主目標/敵波/停止・速度」に絞る。
- 下部は選択部隊1件、部隊アイコン列、命令群、速度操作の固定Dockにする。
- 旅団の長文カード、予測、監査、敵指揮詳細は選択時のdetail drawerへ送る。
- 右側に出す情報は、常時表示の要約と、クリック時の詳細を分離する。

### P1 — 軍の動きの最小表現

画像生成より先に、状態と動きの契約を定める。

| 状態 | 画面上で最低限見えるもの |
|---|---|
| 待機/保持 | 隊列の向き、旗、保持線 |
| 移動 | 進行方向、短い位置補間、移動経路 |
| 射撃 | 一瞬の発砲/煙、対象との射線、再装填状態 |
| 被弾/圧迫 | 隊列の乱れ、警報、士気/戦線圧の変化 |
| 突破/接敵 | 敵群の軸、交戦線、目標への圧力 |
| 後退/崩壊 | 後退方向、隊列の崩れ、施設/目標喪失の影響 |
| 築城/修理 | 工兵の移動、施設耐久の増加、作業中表示 |

### P2 — 局所アセット

P0/P1のレイアウトと動きが固定された後、画像生成を局所投入する。

1. 戦術地図の地形タイル/地図テクスチャ。
2. 自軍/敵軍の小型隊列トークン。歩兵、砲兵、工兵、指揮官、敵主力。
3. 旗、勝利点、補給点、視界点、塹壕、補給所、野戦病院。
4. 発砲煙、着弾、土煙、火災、撤退の短いエフェクト。
5. 選択状態/警戒状態/崩壊状態の差分。

大きな一枚絵を先に置くと、部隊位置・戦線・クリック領域を隠す危険がある。生成画像は背景全面ではなく、地図を読ませる小部品として使う。

## 次に再開するときの検証順

1. UGCW風のbattle viewport wireframeをコード変更前に固定する。
2. 現状のロジックをそのwireframeへ写すためのB契約（save/reload、seed/replayまたはevent/effect log）を定義する。
3. 1280x720で、待機→移動→射撃→敵圧→後退→戦果の短い録画を採る。
4. その録画で、軍が「動いた」だけでなく「なぜ動き、何が変わったか」が読めるか判定する。
5. それから局所アセットを生成し、console error 0、broken image 0を再確認する。

この順番では、戦略mapと戦術mapの分離、主戦場省略不可、小作戦のみauto-resolve、製品版/DLC1規模を維持する。コンテンツ量産は行わない。
