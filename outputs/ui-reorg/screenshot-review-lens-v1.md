# Takawasi Game スクリーンショットレビュー観点 v1

## 結論

「右側の情報欄が妙に長い」は、次のUI再編で明示的に採点する。`horizontal overflow: false` は「切れていない」ことしか証明せず、限られた解像度の機能面積を有効利用できていることは証明しない。

右側の面は、幅だけでなく次を一組で見る。

- 主作業面を何%占有しているか
- 主作業面と同じ強さで視線を奪っていないか
- 長文・履歴・詳細が主操作を下へ押し下げていないか
- 補助情報が要約・折りたたみ・選択時詳細へ分解されているか
- 右面が長くても、その長さが判断のための情報量として機能しているか

## 0. 撮影条件を先に固定する

スクリーンショットの前に、次を記録する。指定解像度と実際のCSS viewportを混同しない。

| 記録 | 例 | 意味 |
|---|---|---|
| target viewport | 1280x720 | QAで要求した基準 |
| actual CSS viewport | `window.innerWidth` / `innerHeight` | ブラウザが実際にレイアウト計算した幅・高さ |
| screenshot pixel size | PNG/JPEGの実寸 | 画像ファイルの寸法。CSS幅とは別物 |
| DPR/zoom | 取得できる場合 | 文字や画像の見かけの倍率 |
| route/state | Theater第10ターン等 | 同じ画面名でも情報量が変わるため |
| scroll position | top / 指定位置 | ファーストビュー判定の再現条件 |

今回のlocal再走では、画像ファイルが1280x720でもDOMのCSS viewportが3878x2181として返るケースを確認した。これは1280px desktop QAの証明には使わず、今後は上表の3寸法を別々に証跡へ残す。

## 1. 面積の使い方

### 横方向

1. 画面の主目的を一文で言い、主作業面を決める。
2. 主作業面が最も広い列になっているかを見る。
3. 右面が補助情報だけなら、主作業面より強い背景・色・文字量を持たせない。
4. 余白が大きい場合は、右面を無理に長くせず、主作業面の地図・戦線・部隊配置へ戻す。
5. カードのintrinsic width、長い日本語ラベル、固定min-widthが列幅を奪っていないかDOMでも確認する。

初期の観測目安は、1280x720のdesktopで主作業面を概ね半分以上確保し、補助面は各4分の1前後を超えて主役化しないこと。ただし、Battleの戦術mapやDeploymentの配置mapなど、画面目的によって配分を変え、数字だけで合否を決めない。

### 縦方向

- header、画面タイトル、主判断、主操作が最初のviewportに入るか
- 右面の履歴・診断・候補一覧が、主操作より先に長く続いていないか
- 1つの情報列を延々と伸ばす代わりに、要約→詳細の段階化ができているか
- 内側scrollを増やす場合、何を固定し何を読む領域にするかが明示されているか
- blankな余白と、読めないほど詰めた密度を同じ「レイアウト問題」として扱わないか

「右欄が長い」は、単に高さを削る問題ではない。右欄の長さが主作業の判断を遅らせるなら、summary、stickyな最小状態、accordion、選択時detailへ分ける。義務情報を削除したり、画像へ置換したりはしない。

## 2. 情報階層の順番

スクリーンショットを次の順に読む。下位の情報が上位を押しのけたら黄色以上とする。

1. 画面名・戦略ターン・現在状態
2. この画面で人間が決める主判断
3. 主戦場または主作業面
4. 直後に使う操作・警報・戻る操作
5. 判断を支える資源、敵情、地形、履歴
6. 詳細診断、候補、ログ、過去履歴

画像はこの順序を強める視覚アンカーであり、文字・操作・義務情報の代替ではない。

## 3. 画面別の観測点

| 画面 | 主作業面 | 右側/補助面の判定 |
|---|---|---|
| Theater | 5層戦略mapと今ターン主戦場 | 小任務は主戦場を押し下げず、自動解決可能の境界を明示する |
| Camp | 軍団・師団・旅団の編成判断 | 旅団詳細は選択対象の説明。全履歴を常時展開して編成面を下へ追いやらない |
| Deployment | 出撃操作と配置map/部隊選抜 | 敵情・戦線診断は配置操作より後ろに埋めない。右面は戦場briefとして短く要約する |
| Battle | 戦術mapと即時指揮 | 警報rail・目標・部隊詳細がmapをDOM下方へ押し出さない。主戦場は省略不可 |
| After Action | 戦果の意味と次ターン遷移 | 損耗詳細は戦果要約と次操作の後に置き、次ターンへ持ち越す値を右面で比較可能にする |
| 次ターン | 直前報告、資源変化、次の戦略判断 | 直前報告が長い場合も要約と全文参照を分け、次の主戦場を見失わせない |

## 4. 画面ごとの最小証跡

各画面で、PNG/JPEGと一緒に次を残す。

- target viewport / actual CSS viewport / screenshot pixel size
- 主作業面の左・上・幅・高さ
- 右側補助面の左・幅・高さ、および主作業面との比率
- 主操作のtop座標、ファーストビュー内か
- 主戦場mapのtop座標、ファーストビュー内か
- `scrollWidth` と `innerWidth`、`scrollHeight` と画面目的に対する妥当性
- console error 0、broken image 0
- 前waveとの差分: 何が広がり、何が押し下がり、何が読めるようになったか

## 5. 判定ラベル

- 緑: 主判断・主操作・主戦場が最初のviewportで理解でき、補助面が主役を奪わない
- 黄: overflowはないが、右面の長さ、空白、重複、内側scrollが主作業を遅らせる
- 赤: 主操作/主戦場が下に埋もれる、切れる、console errorまたはbroken imageがある

黄色を「見た目の好み」として流さず、解像度内の機能面積の損失として記録する。画像生成を追加する前に、この面積・順序・密度を再計測する。
